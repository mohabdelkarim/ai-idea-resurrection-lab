# RFC: add `--build-arg` for `docker compose up --build`

## Summary

This RFC proposes the addition of a `--build-arg` option to the `docker compose up --build` command. This enhancement will allow users to pass build arguments directly to the `docker build` command, streamlining the process of building and starting services with customized images.

## Motivation

Currently, when using `docker compose up --build`, users have to build images separately using `docker compose build` or `docker build` with `--build-arg` and then start the services with `docker compose up`. This process is cumbersome and prone to errors, especially when dealing with multiple services and complex build arguments.

The proposed feature will simplify the workflow by enabling users to pass build arguments directly to `docker compose up --build`, ensuring that images are built with the specified arguments and services are started in a single command.

## Detailed Design

The implementation involves modifying the `docker compose up` command to accept `--build-arg` and pass it to the underlying `docker build` command. This will be achieved by:

1. Extending the `docker compose up` command to accept `--build-arg` option.
2. Parsing the `--build-arg` values and storing them in a format compatible with the BuildKit API.
3. Integrating the BuildKit API to build images with the provided build arguments.
4. Ensuring seamless user experience by maintaining the existing functionality of `docker compose up --build` and adding the new `--build-arg` capability.

The design will focus on:

* Maintaining compatibility with existing `docker compose up` and `docker build` commands.
* Handling multiple `--build-arg` values for different services.
* Providing clear documentation and examples for users.

## Drawbacks

The potential drawbacks of this implementation include:

* Additional complexity in the `docker compose up` command.
* Possible performance impact due to the integration of BuildKit API.
* Potential issues with handling multiple `--build-arg` values for different services.

## Alternatives

Alternatives to this implementation include:

* Continuing to use the existing workaround of running `docker compose build` and `docker compose up` separately.
* Using a separate command, such as `docker compose build-up`, to build and start services.

However, these alternatives do not provide the same level of convenience and streamlined workflow as the proposed implementation.
