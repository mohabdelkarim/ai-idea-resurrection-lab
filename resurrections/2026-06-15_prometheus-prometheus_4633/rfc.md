# RFC: Monitoring kubernetes with prometheus from outside of k8s cluster.

Summary

--------

This RFC proposes a solution to enable Prometheus to monitor Kubernetes clusters from outside the cluster securely and efficiently. The proposed solution involves developing a custom Kubernetes API proxy using Go, which would handle authentication, discovery, and metric scraping.



Motivation

-----------

Monitoring Kubernetes clusters from outside the cluster is crucial for maintaining the health and performance of applications. However, due to the complexity of implementing a secure and scalable solution, this issue was previously abandoned. With advancements in Kubernetes APIs and the maturity of Go-based Kubernetes client libraries, a more straightforward and secure solution can be implemented. The growing need for centralized monitoring solutions makes this issue more pressing.



Detailed Design

----------------

The proposed solution involves the following components:

1. A custom Kubernetes API proxy written in Go, which would be deployed within the Kubernetes cluster.

2. The proxy would utilize Kubernetes' client libraries to interact with the API server, ensuring secure and efficient communication.

3. Prometheus would be configured to scrape metrics from the proxy.

4. The proxy would handle authentication, discovery, and metric scraping.

5. The solution would utilize Kubernetes' built-in authentication and authorization features to ensure secure communication between the proxy and the API server.



The proxy would be designed to:

- Handle authentication with the Kubernetes API server using service accounts and roles.

- Discover available metrics and scrape them from the API server.

- Provide a Prometheus-compatible endpoint for metric scraping.



Drawbacks

----------

The proposed solution has the following drawbacks:

1. Additional complexity: The solution requires developing and deploying a custom proxy, which adds complexity to the existing Prometheus and Kubernetes setup.

2. Potential performance impact: The proxy may introduce additional latency and overhead, potentially impacting the performance of the monitoring solution.



Alternatives

------------

Alternative solutions considered include:

1. Using existing Kubernetes metrics APIs, such as the metrics.k8s.io API, which provides a limited set of metrics.

2. Implementing a sidecar container within each pod to scrape metrics, which would require significant changes to the existing pod configuration.



Unresolved Questions

--------------------

The following questions remain unresolved:

1. How would the proxy handle scaling and high availability in a large Kubernetes cluster?

2. What are the potential security implications of deploying the proxy within the Kubernetes cluster?

3. How would the solution handle future changes to the Kubernetes API and Prometheus?
