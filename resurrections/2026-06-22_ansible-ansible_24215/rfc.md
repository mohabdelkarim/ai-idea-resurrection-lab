# RFC: Increase verbosity on a per task basis

### 1. Summary

This RFC proposes to enhance Ansible's verbosity control by introducing a per-task verbosity setting. The goal is to allow users to increase verbosity on a per-task basis, providing more detailed output for specific tasks in a playbook while maintaining the global verbosity setting for other tasks.

### 2. Motivation

Currently, Ansible's verbosity is controlled globally through command-line options or configuration files. This can lead to overwhelming output when a playbook has many tasks, making it difficult to diagnose issues specific to certain tasks. By implementing per-task verbosity, users can focus on detailed output for tasks that require it, improving debugging efficiency and reducing noise.

### 3. Detailed Design

To implement per-task verbosity, we will:

1. **Introduce a new task-level keyword**: Add a 'verbosity' keyword to tasks, accepting an integer value (similar to the global verbosity setting).
2. **Modify the task executor**: Update the task executor to recognize and handle the new 'verbosity' keyword. When a task with a specified verbosity is executed, the executor will temporarily override the global verbosity setting with the task-specific value.
3. **Utilize existing logging framework**: Leverage Ansible's existing logging mechanisms to control output based on the task-specific verbosity setting.
4. **Develop a custom verbosity plugin**: Create a plugin that allows users to dynamically control verbosity for tasks, including setting verbosity levels based on task conditions or variables.

Example of the new task keyword:

yaml
- name: Example task with increased verbosity
  command: /bin/example
  verbosity: 3
### 4. Drawbacks

1. **Complexity**: Implementing per-task verbosity adds complexity to the task executor and logging mechanisms.
2. **Potential for confusion**: Users may find it confusing to manage multiple verbosity levels across tasks and playbooks.
3. **Performance**: Increased logging verbosity can impact performance, particularly for large playbooks.

### 5. Alternatives

1. **Global verbosity only**: Maintain the current global verbosity setting, which may not provide sufficient flexibility for users needing to debug specific tasks.
2. **Playbook-level verbosity**: Implement verbosity control at the play level instead of the task level, which might not offer enough granularity.

### 6. Unresolved Questions

1. **Verbosity inheritance**: How will verbosity be inherited from parent tasks or blocks? Will it be explicitly set or merged with parent settings?
2. **Interaction with existing logging**: How will per-task verbosity interact with existing logging mechanisms, such as the 'no_log' directive and custom logging plugins?
3. **User interface**: What user interface changes are necessary to support per-task verbosity, including command-line options, configuration settings, and documentation updates?
