# RFC: Better support for sidecar containers in batch jobs

### Summary

This RFC proposes an enhancement to the Kubernetes Job API to better support sidecar containers in batch jobs. The goal is to provide a native way to define and manage Jobs that include sidecar containers, which should terminate gracefully once the main container completes its task.

### Motivation

Batch jobs often require sidecar containers to perform complementary tasks such as logging, monitoring, or cleanup. Currently, Kubernetes Jobs do not differentiate between main containers and sidecar containers, leading to complexities in managing the lifecycle of such pods. The existing workaround involves modifying the sidecar container to detect the main container's termination, which is not ideal. A native solution would simplify the management and enhance the efficiency of batch jobs.

### Detailed Design

The proposed solution involves introducing a new field in the Job API, `mainContainers`, which specifies the names of the main container(s) in the pod. The Job controller will monitor the status of these main containers. Once all main containers have completed (i.e., terminated with a successful exit code or completed their task), the Job will consider itself complete, and the pod will be terminated gracefully, regardless of the status of sidecar containers.

To implement this:

1. **API Change**: Add a `mainContainers` field to the Job spec. This field will be a list of strings, where each string is the name of a container in the pod spec.

2. **Job Controller Logic**: Update the Job controller to watch for the completion of the specified main containers. This can be achieved by enhancing the existing logic that checks pod conditions and exit codes.

3. **kubectl Enhancements**: Extend `kubectl` to provide easier management of Jobs with sidecars, including commands to list Jobs with their main containers and status.

4. **Documentation and Examples**: Provide clear documentation and examples on how to use the new `mainContainers` field in Jobs, including best practices for defining sidecar containers.

### Drawbacks

1. **Complexity in Pod Spec**: Adding another field to the Job spec could add complexity for users who are not familiar with the nuances of Job and Pod lifecycles.

2. **Backward Compatibility**: Existing Jobs might need adjustments to work seamlessly with the new feature, which could involve downtime or additional maintenance.

3. **Potential for Misuse**: If not properly documented, users might misuse the `mainContainers` field, leading to unintended behavior in their batch jobs.

### Alternatives

1. **Pod Lifecycle Hooks**: Rely on Pod lifecycle hooks (e.g., `preStop`, `postStart`) for sidecar containers to detect the main container's termination. However, this approach requires custom code in sidecar containers.

2. **Init Containers**: Using init containers for setup tasks is a current best practice, but they do not directly address the requirement for sidecar containers to run alongside and after main containers.

### Unresolved Questions

1. **Handling Failure**: How should the Job be marked if one of the main containers fails (e.g., with a non-zero exit code)? Should it be considered failed, or should it wait for all main containers to complete successfully?

2. **Default Behavior**: What should be the default behavior if the `mainContainers` field is not specified? Should the current Job behavior be considered the default, where the Job waits for all containers to terminate?

3. **Integration with Existing Tools**: How will existing tools and scripts that manage Jobs need to be updated to work with the new `mainContainers` field?
