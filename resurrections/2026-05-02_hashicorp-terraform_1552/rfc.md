# RFC: aws: Allow rolling updates for ASGs

## Summary
This RFC proposes the introduction of a rolling update mechanism for AWS Auto Scaling groups in Terraform, allowing users to define a custom update policy with parameters such as batch size, pause time, and suspension of processes.

## Motivation
The current Terraform implementation for AWS Auto Scaling groups lacks support for rolling updates, which can lead to downtime and disruptions during deployment. By introducing a rolling update mechanism, users can ensure a smoother and more controlled update process, reducing the risk of errors and improving overall system reliability.

## Detailed Design
The proposed design involves creating a custom 'rolling_update_policy' resource that can be attached to an 'aws_autoscaling_group' resource. This policy would define parameters such as 'max_batch_size', 'min_instances_in_service', 'pause_time', and 'suspend_processes', which would be used to control the update process. The 'aws_autoscaling_group' resource would then use this policy to orchestrate the rolling update, leveraging the AWS Auto Scaling API to replace instances with updated launch configurations.

## Drawbacks
One potential drawback of this proposal is the added complexity of the Terraform configuration, which could lead to increased errors and difficulties in debugging. Additionally, the introduction of a new resource and update mechanism may require significant changes to the existing Terraform codebase, which could be time-consuming and resource-intensive.

## Alternatives
One alternative to this proposal is to use an external tool or service to manage the rolling update process, such as AWS CloudFormation or a third-party deployment tool. However, this approach would require additional setup and configuration, and may not provide the same level of integration and control as a native Terraform solution.

## Unresolved Questions
One unresolved question is how to handle errors and failures during the rolling update process, such as instances that fail to launch or terminate. Additionally, there may be questions around the optimal values for the update policy parameters, such as batch size and pause time, which may vary depending on the specific use case and deployment requirements.
