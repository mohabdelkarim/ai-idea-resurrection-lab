# RFC: [Feature request] Email templating in Alert engine

## Summary

This RFC proposes the implementation of an email templating engine in Grafana's Alert engine. The goal is to enable users to customize their alert notifications using a templating language, making it easier to integrate with various tools and workflows.

## Motivation

Grafana's Alert engine is a powerful tool for monitoring and alerting, but it currently lacks the ability to customize email notifications. Users have requested this feature for years, and with the advancements in templating engines and notification systems, it's now feasible to implement a robust email templating engine. Customizable notifications will improve the overall user experience, making it easier to integrate Grafana with various tools and workflows.

## Detailed Design

The proposed design involves creating a templating engine that integrates with Grafana's existing alerting and notification systems. The templating engine will be built using Go's text/template package, which provides a simple and efficient way to render templates.

The design consists of the following components:

*   TemplateRenderer: A class responsible for rendering templates using the text/template package.
*   NotificationService: A class that handles sending templated notifications.

The workflow will be as follows:

1.  Users create a template for their alert notifications using Grafana's templating language.
2.  The TemplateRenderer class renders the template using the alert data.
3.  The rendered template is then passed to the NotificationService class, which sends the templated notification.

The templating language will support the following features:

*   Variables: Users can access alert data using variables, such as `{{ .Alert.Name }}`.
*   Conditionals: Users can use conditionals, such as `{{ if .Alert.Firing }}`.
*   Loops: Users can use loops, such as `{{ range .Alert.Annotations }}`.

## Drawbacks

The main drawback of this proposal is the added complexity to Grafana's Alert engine. The templating engine will require additional resources and maintenance, which may impact performance.

Another potential drawback is the learning curve for users. The templating language will require users to learn a new syntax and understand how to access alert data.

## Alternatives

One alternative to this proposal is to use an existing templating engine, such as Jinja2 or Mustache. However, integrating an external templating engine may add additional complexity and dependencies.

Another alternative is to use a notification-as-a-service solution, such as PagerDuty or OpsGenie. However, these solutions may require additional setup and configuration, and may not provide the same level of customization as a built-in templating engine.

## Unresolved Questions

The following questions remain unresolved:

*   What is the optimal syntax for the templating language?
*   How will users be able to access alert data in their templates?
*   What are the performance implications of rendering templates for each alert notification?
*   How will the templating engine handle errors and edge cases?

These questions will need to be addressed during the implementation phase of the proposal.
