# RFC: feature request: looping over blocks

## 1. Summary

This RFC proposes the implementation of a feature to loop over a block of tasks in Ansible. The goal is to enhance the dynamic and reusable capabilities of Ansible playbooks by allowing users to iterate over a block of tasks, making it easier to manage complex automation workflows.

## 2. Motivation

The Ansible community has been requesting this feature for several years, with the initial issue opened in 2015. The need for looping over blocks arises from the increasing complexity of automation tasks and the desire for more dynamic playbook structures. Currently, users have to rely on workarounds such as using the `loop` module with a single task or using `include_tasks` with a loop, which can lead to less readable and maintainable playbooks.

## 3. Detailed Design

The proposed design involves integrating the existing `loop` module with block structures. This can be achieved through a new module or a modification to the existing block syntax to support looping. The design should ensure that:

* Loop variables are accessible within the block
* The loop can be controlled using Ansible's existing loop control mechanisms (e.g., `loop_control`)
* The loop can iterate over a list of items, similar to the existing `loop` module

A possible implementation could involve creating a new class, `LoopableBlock`, that inherits from the existing `Block` class and adds loop functionality. This class would need to handle loop variable scoping, loop control, and iteration over the list of items.

## 4. Drawbacks

The main drawbacks of this feature are:

* Potential performance impact due to the added complexity of loop handling
* Possible introduction of new errors or edge cases, such as infinite loops or loop variable collisions
* The need for additional documentation and community education to ensure users understand the feature and its limitations

## 5. Alternatives

Alternative solutions to looping over blocks include:

* Using the existing `loop` module with a single task, which can lead to less readable playbooks
* Using `include_tasks` with a loop, which can lead to additional complexity and performance overhead
* Implementing a custom module or plugin to achieve the desired functionality, which can be more difficult to maintain and support

## 6. Unresolved Questions

The following questions remain to be addressed:

* How to handle loop variable scoping and collisions?
* What is the best way to integrate loop control mechanisms with block structures?
* How to ensure that the feature is properly documented and communicated to the community?
* What are the potential performance implications of this feature, and how can they be mitigated?
