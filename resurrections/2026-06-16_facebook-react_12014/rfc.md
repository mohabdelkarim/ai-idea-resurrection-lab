# RFC: Have Fragments support dangerouslySetInnerHTML

# Summary

This RFC proposes to enable the `dangerouslySetInnerHTML` property on React Fragments. This feature will allow developers to inject HTML content directly into a Fragment, providing more flexibility in rendering dynamic content.




# Motivation

React Fragments are a lightweight way to group children without adding extra nodes to the DOM. However, they currently do not support the `dangerouslySetInnerHTML` property, which is available on other React components. This limitation makes it difficult to inject HTML content into a Fragment. With the growing use of server-side rendering and static site generation with React, the demand for flexible HTML injection methods has increased.




# Detailed Design

To implement this feature, we will add a check for `dangerouslySetInnerHTML` on Fragment components. If present, React will unwrap the Fragment and directly inject the HTML into the DOM, bypassing the normal Fragment rendering logic. This will involve modifications to React's reconciliation algorithm and the Fragment component's implementation.




The injected HTML will be sanitized using a library like DOMPurify to prevent security vulnerabilities. We will also ensure that the implementation follows React's existing security guidelines.




Here is an example of how this feature could be used:

jsx
import React from 'react';

function MyComponent() {
  const html = '<div>Injected HTML</div>';
  return (
    <>
      <div>Before injection</div>
      <span dangerouslySetInnerHTML={{ __html: html }} />
      <div>After injection</div>
    </>
  );
}



# Drawbacks

The main drawback of this feature is the potential security risk associated with injecting arbitrary HTML into the DOM. However, with proper sanitization and escaping, this risk can be mitigated.




Another potential drawback is the impact on performance. Injecting HTML directly into the DOM can be faster than rendering React components, but it may also bypass some of React's optimizations.




# Alternatives

One alternative to this feature is to use a library like React-HTML-Parser to parse the HTML content and render it as React components. However, this approach can be cumbersome and may not provide the same level of flexibility as injecting HTML directly.




Another alternative is to use a different component, such as a `div` or `span`, and use the `dangerouslySetInnerHTML` property on that component. However, this approach may not be suitable for all use cases, especially when a Fragment is the only suitable container.




# Unresolved Questions

* How will we handle the case where the Fragment is used as a child of another component that also uses `dangerouslySetInnerHTML`?
