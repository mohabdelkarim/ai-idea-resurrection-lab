# RFC: Change Request: Show a reminder when errors are suppressed

### 1. Summary

ESLint should optionally show reminders when errors are suppressed to improve visibility of tech debt.

This RFC proposes an enhancement to ESLint that provides users with optional reminders when errors are suppressed. The goal is to increase visibility of technical debt by gently reminding users of suppressed errors, which can help in prioritizing and addressing these issues over time.

### 2. Motivation

Currently, ESLint suppresses errors silently when configured to do so. While this allows for a smooth development workflow, it can also lead to overlooking technical debt. As modern development tools continue to evolve, there's an opportunity to enhance the ESLint experience by providing reminders of suppressed errors. This feature aligns with current trends in code analysis and tech debt management.

### 3. Detailed Design

The proposed solution involves introducing a new feature in ESLint that optionally reminds users of suppressed errors. This can be achieved by:

- Introducing a new configuration option or command-line flag, e.g., `--remind-suppressions` or `rules.remindSuppressions`, that users can enable to receive reminders.
- Modifying the `LintResult` object to include metadata about suppressed errors.
- Creating a `SuppressionsReminder` class that processes lint results and emits informational diagnostics when suppressed errors are detected.

The reminders should not interfere with existing error reporting or affect exit codes. They should be opt-in to preserve the silent suppression flow for users who prefer it.

### 4. Drawbacks

- **Noise**: If not implemented carefully, reminders could add unnecessary noise to the development workflow, potentially leading to user frustration.
- **Configuration Complexity**: Adding another configuration option or flag could contribute to configuration fatigue, especially if not clearly documented or if it interacts with existing configurations in complex ways.

### 5. Alternatives

- **Integrate with existing tools**: Instead of building a reminder feature directly into ESLint, consider integrating with external tools or plugins that specialize in tech debt management and code analysis.
- **Enhance documentation**: Focus on improving documentation around error suppression to help users understand the implications and best practices for suppressing errors.

### 6. Unresolved Questions

- **User opt-in mechanisms**: How will users opt-in to this feature? Should it be a configuration option, a command-line flag, or both?
- **Reminder frequency and format**: What is the best format and frequency for reminders? Should they be integrated into the ESLint output, or could they be displayed as pop-ups in an IDE?
- **Interaction with existing configurations**: How will this feature interact with existing ESLint configurations, especially those that suppress errors?
