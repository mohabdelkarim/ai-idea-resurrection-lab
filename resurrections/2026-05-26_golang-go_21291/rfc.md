# RFC: proposal: Go 2: remove bare return

## Summary

This RFC proposes the removal of bare returns in Go 2, a feature that has been a subject of controversy and discussion in the Go community. With modern tooling and a more open community, we believe the time is ripe to revisit this proposal. The goal is to improve code readability and reduce errors by requiring named returns.

## Motivation

Bare returns in Go allow a function to return values without specifying the return parameters. While this feature was intended to simplify code, it has led to confusion and errors, particularly in complex functions with multiple return values. The Go team has previously discussed removing bare returns, but the proposal was abandoned due to resistance from the community.

The landscape has changed since then. The introduction of Go 1.18's generics feature and the upcoming Go 1.20 release have created a more favorable environment for this proposal. Modern tooling, such as gofmt and go vet, can simplify the transition by automatically converting bare returns to named returns.

## Detailed Design

The proposed design involves the following steps:

1. **Linter Rule**: Introduce a new linter rule that flags bare returns as errors. This rule will be integrated into the existing go vet tool.
2. **gofmt Option**: Add a gofmt option to automatically convert bare returns to named returns. This will help developers transition their code to the new syntax.
3. **Language Specification Update**: Update the language specification to reflect the removal of bare returns.
4. **Documentation and Tooling**: Provide documentation to help developers adapt to the new syntax and offer tools to convert existing code.

The rollout will be gradual, allowing developers to opt-in to the new behavior. The Go team will monitor the transition and provide support as needed.

## Drawbacks

The main drawback of this proposal is the disruption it may cause to existing codebases. However, with modern tooling and a gradual rollout, we believe the benefits outweigh the costs. Another potential drawback is the initial resistance from developers who have grown accustomed to bare returns.

## Alternatives

An alternative approach could be to introduce a new warning for bare returns, rather than removing them entirely. However, this approach may not provide enough incentive for developers to change their code. Another alternative is to introduce a new syntax for bare returns, but this may add complexity to the language.
