# RFC: URL resolution case sensitivity is inconsistent

### 1. Summary

Inconsistent URL resolution case sensitivity in Next.js leads to unexpected 404 errors. This RFC proposes a solution to implement case-insensitive URL resolution while maintaining performance and SEO best practices.

### 2. Motivation

The current implementation of URL resolution in Next.js is case-sensitive, which can lead to unexpected 404 errors. For example, if a page is located at `/about`, requesting `/About` or `/ABOUT` will result in a 404 error. This inconsistency can negatively impact user experience and SEO.

The goal of this RFC is to propose a solution that provides case-insensitive URL resolution while maintaining performance and SEO best practices.

### 3. Detailed Design

The proposed solution involves implementing a middleware layer that checks for case-insensitive matches in file system routes. If a match is found, a 301 redirect is issued to the correct case.

The implementation would involve creating a custom middleware function that uses the `next/server` API to handle redirects and rewrites. The middleware function would:

1. Check if the requested URL matches a file system route.
2. If a match is found, check if the case of the requested URL matches the case of the file system route.
3. If the cases do not match, issue a 301 redirect to the correct case.

The middleware function would be executed on every request, ensuring that all URLs are resolved consistently.

To ensure performance, the middleware function would be optimized to minimize overhead. This could be achieved by:

1. Caching frequently accessed routes.
2. Using a efficient data structure to store file system routes.

### 4. Drawbacks

The proposed solution has the following drawbacks:

1. **Performance overhead**: Implementing a middleware layer could introduce performance overhead, particularly if the function is not optimized.
2. **Redirects and rewrites**: Implementing case-insensitive URL resolution could lead to additional redirects and rewrites, which could impact SEO.

However, these drawbacks can be mitigated by:

1. Optimizing the middleware function to minimize overhead.
2. Implementing redirects and rewrites in a way that minimizes impact on SEO.

### 5. Alternatives

The following alternatives were considered:

1. **Case-sensitive URL resolution**: Maintaining the current case-sensitive URL resolution behavior.
2. **Case-insensitive URL resolution using rewrites**: Implementing case-insensitive URL resolution using rewrites instead of redirects.

However, these alternatives were rejected because:

1. **Case-sensitive URL resolution**: Does not provide a consistent user experience and can negatively impact SEO.
2. **Case-insensitive URL resolution using rewrites**: May not provide the same level of SEO benefits as redirects.
