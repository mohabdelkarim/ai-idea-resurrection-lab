# RFC: Feature Request: add support for error bars in graphs

### Summary

This RFC proposes adding support for configurable error bars in Grafana graphs. Error bars will provide a visual representation of the uncertainty or variability in the data being displayed, enhancing the overall data visualization and monitoring capabilities of Grafana.




### Motivation

The growing demand for data visualization and monitoring capabilities in Grafana necessitates the addition of error bars. Error bars are a common feature in scientific and technical data visualization, allowing users to understand the reliability and variability of the data. With the advancements in Grafana's plugin architecture and the introduction of new data processing libraries, it's now feasible to implement error bars. Furthermore, the latest versions of InfluxDB and other data sources provide better support for statistical data.




### Detailed Design

To implement error bars, we will leverage Grafana's existing panel plugin architecture and utilize libraries like uPlot or D3.js for rendering. The graph panel will be extended to accept additional configuration options for error bars, such as:

*   Type of interval (e.g., standard deviation, percentile)
*   Number of samples to use
*   Option to display error bars as a percentage or absolute value

The data processing layer will be modified to extract the required statistical data from the data sources. This will involve:

*   Adding support for statistical data processing in the data query layer
*   Integrating with data sources that provide statistical data, such as InfluxDB

The error bars will be rendered as part of the graph panel, with options for customization, such as color, line style, and label formatting.




### Drawbacks

The main drawbacks of this proposal are:

*   Additional complexity in the graph panel configuration
*   Potential performance impact due to increased data processing and rendering requirements
*   Limited support for certain data sources or data types

However, these drawbacks can be mitigated by:

*   Providing a simple and intuitive configuration interface
*   Optimizing the data processing and rendering performance
*   Documenting the limitations and requirements for data sources and data types




### Alternatives

Alternative solutions considered include:

*   Using external plugins or libraries to provide error bar functionality
*   Implementing error bars as a separate panel type

However, these alternatives are not preferred because:

*   External plugins or libraries may not be well-integrated with the Grafana ecosystem
*   A separate panel type may lead to duplication of effort and confusion for users




### Unresolved Questions

The following questions remain to be resolved:

*   What is the optimal way to handle error bars for data sources that do not provide statistical data?
*   How will error bars be handled for data with non-numeric or non-statistical values?
*   What are the performance implications of rendering error bars for large datasets?

These questions will be addressed through further research, testing, and iteration during the implementation phase.
