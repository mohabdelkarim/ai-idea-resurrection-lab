# RFC: proposal: spec: lazy values

Summary

    --------

    This RFC proposes the introduction of a lazy value mechanism in Go, allowing for the delay of evaluation of expressions until their values are actually needed. This feature aims to improve performance by avoiding unnecessary computations and could be particularly beneficial when used with expensive operations or in scenarios where the result of an expression might not always be required.


    Motivation

    ---------

    The motivation behind this proposal is to provide Go developers with a tool to write more efficient code. With the increasing use of functional programming patterns and the introduction of generics in Go 1.22, there's a growing interest in features that support lazy evaluation. Additionally, the potential for lazy logging and error handling could significantly benefit from this feature, making it easier to implement logging and error reporting mechanisms that only incur a performance cost when actually needed.


    Detailed Design

    --------------

    The proposed implementation involves introducing a new type, 'Lazy[T any]', which wraps a function that returns a value of type T. This type would have at least two methods:

    - 'Value() T': This method would evaluate the wrapped function and return the result. It would use a mutex to ensure thread safety, preventing multiple goroutines from evaluating the function simultaneously.

    - 'IsEvaluated() bool': This method would allow checking if the value has already been computed, providing a way to avoid unnecessary evaluations.

    

    Example usage:

    ```go

    lazyValue := Lazy[int]{func() int {

        // Expensive operation

        time.Sleep(1 * time.Second)

        return 42

    }}

    

    fmt.Println(lazyValue.IsEvaluated()) // prints: false

    value := lazyValue.Value()

    fmt.Println(value) // prints: 42

    fmt.Println(lazyValue.IsEvaluated()) // prints: true

    ```

    This design ensures that the expensive operation is only performed when the value is actually needed, demonstrating how lazy values can improve performance.


    Drawbacks

    ---------

    While the proposed lazy value mechanism offers several benefits, there are also potential drawbacks to consider:

    - **Added Complexity**: Introducing a new type and methods could add complexity to the language, potentially making it harder for newcomers to learn.

    - **Performance Overhead**: Although the goal is to improve performance, the implementation of lazy values would itself incur a small performance overhead due to the use of mutexes and function calls.

    - **Debugging Challenges**: The delayed evaluation of expressions could make debugging more difficult, as the code execution flow might be harder to follow.


    Alternatives

    ------------

    Before settling on the 'Lazy[T any]' type, alternative approaches were considered:

    - **Functions as First-Class Citizens**: Go already supports functions as first-class citizens, which can be used to achieve similar outcomes. However, this approach lacks the explicit 'IsEvaluated()' method and thread safety provided by the proposed 'Lazy[T]' type.

    - **Existing Libraries**: There are existing libraries that implement lazy evaluation patterns. However, integrating such patterns directly into the language could provide a more streamlined and efficient solution.


    Unresolved Questions

    -------------------

    - **Integration with Error Handling**: How should errors be handled within the lazy value mechanism? Should the 'Value()' method return an error, or should there be a separate 'Error()' method?

    - **Panic Behavior**: What should happen if the wrapped function panics? Should the panic be propagated, or should it be handled in a specific way by the 'Lazy[T]' type?

    - **Implications for Garbage Collection**: Are there any implications for garbage collection due to the potential for cyclic references or long-lived closures?

    Addressing these questions will be crucial to refining the proposal and ensuring that the lazy value mechanism is both useful and well-integrated into the Go language ecosystem.
