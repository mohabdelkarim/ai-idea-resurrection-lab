# RFC: proposal: Go 2: Have functions auto-implement interfaces with only a single method of that same signature

## Summary

This proposal suggests that the Go language automatically implement single-method interfaces for functions with matching signatures. This feature aims to reduce boilerplate code and make Go programs more concise and expressive.

## Motivation

Currently, Go requires explicit implementation of interfaces, which can lead to verbose code. For example, if we have a single-method interface `Logger` and a function `logMessage`, we need to write an extra struct and implementation block to satisfy the interface.

go
type Logger interface {
    Log(string)
}

func logMessage(s string) {
    // log the message
}

// need to write an extra struct and implementation block
type logger struct{}

func (l *logger) Log(s string) {
    logMessage(s)
}
With this proposal, the `logMessage` function would automatically implement the `Logger` interface, eliminating the need for the extra struct and implementation block.

## Detailed Design

To implement this feature, we propose modifying the Go compiler to automatically generate an interface implementation for a function if it matches the signature of a single-method interface. We will add a new AST node type to represent the interface implementation and modify the type checker to generate the implementation.

The implementation will follow these rules:

*   A function automatically implements a single-method interface if its signature matches the interface method signature.
*   The function and interface method must have the same name, return types, and parameter types.
*   The automatic implementation only applies to single-method interfaces.

## Drawbacks

While this feature aims to reduce boilerplate code, it may also introduce subtle issues:

*   **Backwards compatibility**: Existing code may rely on the current behavior of interfaces, and this change could break existing programs.
*   **Ambiguity**: If multiple interfaces have the same method signature, it's unclear which interface a function should implement.

## Alternatives

Instead of automatically implementing single-method interfaces, we could:

*   **Provide a code generation tool**: Offer a tool that generates the interface implementation code for the developer.
*   **Introduce a new syntax**: Add a new syntax to explicitly indicate that a function implements an interface.
