# RFC: Using scale number to adjust mounted volumes?

### Summary

This RFC proposes the addition of a feature to Docker Compose that allows dynamic volume mounting using scale numbers. The goal is to enable users to mount volumes in a scalable and flexible manner, making it easier to manage complex container deployments.




### Motivation

Currently, Docker Compose requires explicit volume mounts to be defined in the `docker-compose.yml` file. This can become cumbersome when dealing with multiple containers and complex volume mounting scenarios. The proposed feature would enable users to use templating in volume mounts, allowing them to reference environment variables or scale numbers. This would provide more flexibility and scalability in managing container deployments.




### Detailed Design

The proposed feature would involve the following changes:

*   Add a templating engine to Docker Compose that supports referencing environment variables and scale numbers in volume mounts.
*   Modify the Docker Compose configuration file parser to support templating.
*   Update the volume mounting logic to resolve templated volume mounts.

The templating engine would use a syntax similar to existing templating engines, such as `{{ }}` or `${ }}`. Users would be able to reference environment variables and scale numbers using this syntax.

For example:

yml
version: '3'
services:
  web:
    image: nginx
    volumes:
      - ./data:/data/{{scale.web}}:ro
In this example, the volume mount would be scaled based on the `scale.web` value.




### Drawbacks

The proposed feature introduces potential security concerns, such as path traversal attacks. To mitigate this, the implementation would need to include robust error handling and validation of templated volume mounts.

Additionally, the feature could add complexity to the Docker Compose configuration file, potentially making it harder to understand and maintain.




### Alternatives

Alternative solutions to this problem include:

*   Using third-party tools or scripts to manage volume mounts.
*   Implementing a custom solution using Docker Compose's existing features, such as using environment variables to construct volume mounts.

However, these alternatives may not provide the same level of flexibility and scalability as the proposed feature.




### Unresolved Questions

The following questions remain unresolved:
