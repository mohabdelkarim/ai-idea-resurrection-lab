# RFC: Support having plugins as dependencies in shareable config

# Summary

This RFC proposes to enhance ESLint's shareable configuration feature by allowing plugins to be listed as dependencies. This enhancement will provide a seamless user experience when using shareable configs by automatically installing and loading required plugins.

 
# Motivation

Currently, when using a shareable config, users are required to manually install all the plugins that the config depends on. This can be tedious and error-prone, especially for complex configs with many dependencies. By supporting plugins as dependencies in shareable configs, users can simply extend the config without worrying about manually installing plugins.

The motivation behind this proposal is to improve the overall user experience of ESLint by reducing the friction of using shareable configs. With this enhancement, users can easily share and reuse configurations across projects, promoting consistency and best practices.

 
# Detailed Design

To implement this feature, we propose the following:

*   Introduce a new class, `PluginResolver`, responsible for resolving plugins listed in shareable configs.
*   Modify the `ConfigMerger` class to interact with the `PluginResolver` to ensure seamless integration.
*   Update the configuration processing mechanism to automatically install and load plugins listed as dependencies in shareable configs.

The `PluginResolver` will utilize the `require.resolve` function to find plugins, allowing for more flexible plugin management. When a shareable config is extended, ESLint will:

1.  Load the shareable config.
2.  Resolve the plugins listed as dependencies in the config.
3.  Install the plugins if they are not already installed.
4.  Load the plugins.

This design ensures that plugins are properly resolved, installed, and loaded, providing a seamless user experience.

 
# Drawbacks

The main drawback of this proposal is the potential for increased complexity in the ESLint configuration processing mechanism. Additionally, there may be concerns about security and vulnerability management when automatically installing plugins.

However, these drawbacks can be mitigated by:

*   Implementing robust security measures, such as validating plugin sources and ensuring that plugins are properly sandboxed.
*   Providing clear documentation and guidelines for users to manage plugin dependencies.

 
# Alternatives

An alternative approach would be to require users to manually install plugins using a separate command, such as `eslint --install-plugins`. However, this approach would still require users to manually manage plugin dependencies, which is the problem this proposal aims to solve.

Another alternative is to include plugins as part of the shareable config package. However, this approach would lead to bloated packages and make it difficult to manage plugin dependencies.

 
# Unresolved Questions

The following questions remain to be addressed:

*   How to handle plugin conflicts or version mismatches?
*   How to ensure that plugins are properly validated and secured?
*   How to provide clear documentation and guidelines for users to manage plugin dependencies?

These questions will need to be addressed during the implementation phase to ensure that the feature is robust, secure, and user-friendly.
