# Analysis: Support having plugins as dependencies in shareable config

> ESLint shareable configs can now include plugins as dependencies for a seamless user experience.

**Why it will work now:** Supporting plugins as dependencies in shareable configs enhances user experience by minimizing boilerplate.

## Why It Died

The issue failed because ESLint's architecture at the time didn't support plugins as dependencies in shareable configs. The plugin loading mechanism was tightly coupled with the configuration merging process. This limitation forced users to manually install required plugins.

## Why 2026 Changes Everything

With the advancements in ESLint's plugin system and the introduction of more modular designs, it's now feasible to support plugins as dependencies. Specifically, changes in ESLint's configuration processing and plugin loading mechanisms make this feature viable. Additionally, the growth of the ESLint ecosystem and the increasing number of plugins make this enhancement more valuable.

## Modern Architecture

The modern design would involve decoupling plugin loading from configuration merging. We can introduce a new class, `PluginResolver`, responsible for resolving plugins listed in shareable configs. This class would interact with the `ConfigMerger` to ensure seamless integration. The `PluginResolver` would utilize the `require.resolve` function to find plugins, allowing for more flexible plugin management.
