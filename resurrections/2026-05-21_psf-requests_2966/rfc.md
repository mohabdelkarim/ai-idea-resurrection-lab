# RFC: Consider using system trust stores by default in 3.0.0.

Summary

--------

This RFC proposes changing the default behavior of the requests library to use system trust stores for verifying SSL certificates, starting from version 3.0.0. This change aims to improve security and compliance by leveraging the trust stores managed by operating systems and package managers.



Motivation

---------

The requests library currently uses a bundled certificate authority (CA) trust store, certifi, to verify SSL certificates. While this approach ensures broad compatibility and ease of use, it has several drawbacks. System trust stores are updated more frequently by operating systems and package managers, ensuring that the latest security patches and certificate updates are applied. Using system trust stores by default can enhance security and compliance, particularly in environments where certificate management is critical.



Detailed Design

---------------

The proposed design involves the following steps:

1. Check for the presence of a library like certitude on the system.

2. If present, use it to generate the trust store for verifying SSL certificates.

3. If not present, fall back to using certifi's bundled trust store.

4. Implement this logic using the Adapter pattern, allowing for easy swapping of trust store implementations.

5. Provide configuration options to allow users to choose between system trust stores, certifi, or custom trust stores.

6. Ensure seamless integration with existing codebases by maintaining backward compatibility.



Drawbacks

----------

1. **Compatibility Issues**: Relying on system trust stores might introduce compatibility issues on systems with outdated or custom trust stores.

2. **Complexity**: Implementing and testing the integration with various system trust store solutions and certitude-like libraries across different platforms may add complexity.

3. **User Education**: Users may need to be educated about the changes and how to manage system trust stores effectively.



Alternatives

------------

1. **Continue using certifi**: Maintain the status quo and continue using certifi's bundled trust store.

2. **Mandate system trust stores**: Force the use of system trust stores without providing a fallback or configuration options.

3. **Introduce a hybrid approach**: Allow users to choose between system trust stores and certifi on a per-session basis.
