# RFC: Consider making Timeout option required or have a default

## 1. Summary

The requests library currently does not have a default timeout value, which can lead to hanging requests if a timeout is not explicitly set. This RFC proposes adding a default timeout value or making the timeout option required to prevent such issues.

## 2. Motivation

The requests library is a popular and widely-used library for making HTTP requests in Python. However, its current design allows for requests to hang indefinitely if a timeout is not specified. This can lead to problems such as resource leaks, slow performance, and difficulties in debugging. With the growing emphasis on asynchronous programming in Python, making timeouts more explicit aligns with modern practices.

## 4. Drawbacks

One potential drawback of adding a default timeout value is that it may break existing code that relies on the current behavior of requests hanging indefinitely. However, this can be mitigated by providing a clear documentation and a transition guide for users.

Another potential drawback is that a default timeout value may not be suitable for all use cases. For example, some requests may need to run for an extended period, and a default timeout value could interrupt these requests prematurely.

## 5. Alternatives

One alternative to adding a default timeout value or making the timeout option required is to provide a warning when a timeout is not specified. This would inform users that their requests may hang indefinitely but would not change the current behavior.
