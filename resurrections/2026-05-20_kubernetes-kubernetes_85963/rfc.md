# RFC: Add Support to Deny RBAC Rules

# 1. Summary

This RFC proposes the addition of deny rules to Kubernetes Role-Based Access Control (RBAC) to enhance access control and restrict permissions more effectively. The existing RBAC system is primarily designed with allow rules in mind, which can lead to overly permissive access control in certain scenarios. By introducing deny rules, we aim to provide a more fine-grained and flexible access control mechanism.

The proposed design involves introducing a new 'Deny' kind for RBAC rules, which would be evaluated after allow rules. This would utilize existing policy engines like CEL (Common Expression Language) for more flexible rule definitions.





# 2. Motivation

The current RBAC system in Kubernetes only supports allow rules, which can make it difficult to restrict access effectively. For example, when a user has multiple roles bound to them, the union of all allow rules applies, potentially leading to overly permissive access. Deny rules can help mitigate this issue by explicitly denying access to certain resources or actions.

Moreover, with the advancements in Kubernetes' security features, such as the introduction of ValidatingAdmissionPolicies and improved support for granular access control, the groundwork has been laid for incorporating deny rules into RBAC.





# 3. Detailed Design

The proposed design involves the following key components:

*   Introduction of a new 'Deny' kind for RBAC rules
*   Evaluation of deny rules after allow rules
*   Utilization of existing policy engines like CEL for flexible rule definitions
*   Definition of a PolicyEvaluator interface for evaluating allow and deny rules
*   Introduction of AllowRule and DenyRule structs for representing allow and deny rules
*   Integration with admission controllers for seamless policy enforcement

The deny rules would be defined in a similar way to allow rules but with a 'Deny' kind. The evaluation of deny rules would occur after allow rules, ensuring that even if an allow rule grants access, a subsequent deny rule can still deny it.





# 4. Drawbacks

The introduction of deny rules to Kubernetes RBAC comes with several potential drawbacks:

*   **Complexity**: Adding deny rules increases the complexity of the RBAC system, potentially making it harder to understand and manage.
*   **Backwards Compatibility**: Existing workflows and tools may need to be updated to handle deny rules, which could lead to backwards compatibility concerns.
*   **Performance**: Evaluating deny rules after allow rules could potentially impact performance, especially in large-scale deployments.





# 5. Alternatives

Several alternatives have been considered:

*   **Attribute-based access control (ABAC)**: ABAC provides a more fine-grained access control mechanism but can be more complex to implement and manage.
*   **External policy engines**: External policy engines like OPA/Gatekeeper and Kyverno can provide complex policy enforcement but may require additional infrastructure and management.

The proposed design of introducing deny rules to Kubernetes RBAC provides a more straightforward and integrated solution.
