# RFC: Create Categories/Groups for extensions.

## 1. Summary

This RFC proposes the implementation of extension categorization and grouping in Visual Studio Code (VSCode). The goal is to simplify the enablement of extensions for specific workspaces, making it easier for users to manage their extensions and for extension authors to reach their target audience.

The proposed solution involves introducing an 'ExtensionCategories' API, allowing extension authors to declare categories for their extensions, and a 'ExtensionGroups' class to manage these categories and enable/disable extensions based on workspace settings. The UI will integrate with the existing Extensions view, incorporating categorization and grouping.

## 2. Motivation

As the VSCode extension ecosystem has matured, user demands for better extension organization have grown. The current extension management system can be overwhelming, especially for users with a large number of installed extensions. By introducing categorization and grouping, users will be able to easily find and enable extensions relevant to their current workspace, improving their overall experience.

The existence of Workspace Trust and extension enablement per workspace provides a foundation for this feature. Additionally, the advancements in VSCode's extension management, such as the Extensions view and extension recommendations, lay the groundwork for implementing categorization.

## 3. Detailed Design

The proposed solution consists of the following components:

*   **ExtensionCategories API**: Extension authors will be able to declare categories for their extensions using a new API. This API will allow authors to specify a list of categories that their extension belongs to.
*   **ExtensionGroups class**: A new class, 'ExtensionGroups', will be introduced to manage the extension categories and enable/disable extensions based on workspace settings. This class will provide methods for users to create, edit, and delete extension groups.
*   **UI Integration**: The existing Extensions view will be updated to incorporate categorization and grouping. Users will be able to view extensions grouped by category and easily enable/disable extensions for specific workspaces.

The 'ExtensionCategories' API will be designed to be extensible, allowing for new categories to be added in the future. The 'ExtensionGroups' class will utilize the existing extension enablement per workspace feature to enable/disable extensions based on workspace settings.

## 4. Drawbacks

The main drawback of this proposal is the potential complexity added to the extension management system. Extension authors will need to declare categories for their extensions, which may require additional effort. However, this effort will be minimal, as the 'ExtensionCategories' API will be designed to be simple and easy to use.

Another potential drawback is the need for users to learn a new feature. However, the UI will be designed to be intuitive, and users will be able to easily understand how to use the categorization and grouping features.

## 5. Alternatives

One alternative to this proposal is to rely on existing solutions, such as extension recommendations and the Extensions view. However, these solutions do not provide the same level of organization and enablement as categorization and grouping.

Another alternative is to use a tagging system, where users can assign tags to extensions. However, this approach may lead to a proliferation of tags, making it difficult for users to find relevant extensions.
