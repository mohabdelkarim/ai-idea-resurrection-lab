# RFC: Support multiple windows

## Summary

Alacritty currently lacks native support for multiple windows, requiring users to open new instances of the terminal emulator. This RFC proposes the implementation of a `WindowManager` class that handles window creation, management, and destruction, allowing Alacritty to support multiple windows.

## Motivation

The lack of multiple window support in Alacritty has been a long-standing issue, with users requesting this feature for improved workflow and productivity. The current workarounds, such as opening new instances of Alacritty or using tmux, are not ideal and can lead to performance issues. With the advancement of Rust's GUI libraries and cross-platform window management APIs, it's now feasible to implement multiple windows in Alacritty.

## Detailed Design

The implementation of multiple windows in Alacritty will involve the following steps:

* Create a new `WindowManager` class that handles window creation, management, and destruction.
* Utilize the winit API for window management and integrate with Alacritty's existing `Terminal` class.
* Refactor the `Terminal` class to support multiple instances.
* Implement the MVC pattern, with the `WindowManager` acting as the controller.

The `WindowManager` class will be responsible for:

* Creating new windows with a specified configuration (e.g., window size, position, and title).
* Managing window events, such as resize, move, and close.
* Destroying windows when they are closed.

The `Terminal` class will need to be refactored to support multiple instances, with each instance representing a single window.

## Drawbacks

The implementation of multiple windows in Alacritty will have the following drawbacks:

* Increased complexity: The addition of a new `WindowManager` class and the refactoring of the `Terminal` class will increase the complexity of the codebase.
* Performance impact: The creation and management of multiple windows may have a performance impact, particularly if the user opens many windows.

## Alternatives

The following alternatives have been considered:

* Using a library that provides multiple window support, such as tmux or GNU Screen. However, these libraries have their own set of limitations and may not integrate well with Alacritty.
* Implementing a tabbed interface, similar to some other terminal emulators. However, this approach may not provide the same level of flexibility as multiple windows.

## Unresolved Questions

The following questions remain unresolved:

* How will the `WindowManager` class handle window configuration and settings? Will each window have its own configuration, or will there be a global configuration that applies to all windows?
* How will the `Terminal` class handle multiple instances? Will each instance have its own event loop, or will there be a shared event loop that handles events for all instances?
* What are the performance implications of implementing multiple windows, and how can they be mitigated?
