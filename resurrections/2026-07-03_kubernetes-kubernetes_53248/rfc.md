# RFC: kubectl provides a way to find which RoleBinding/ClusterRoleBinding is related to a serviceAccount

# 1. Summary

This RFC proposes the addition of a new kubectl subcommand to find RoleBinding/ClusterRoleBinding related to a serviceAccount. The proposed command will take a service account name and namespace as input, query the Kubernetes API for related role bindings, and output the results.

## 2. Motivation

As Kubernetes adoption continues to grow, so does the complexity of managing access control and security within clusters. Service accounts and role bindings are critical components in securing Kubernetes clusters, but managing them can become cumbersome as the number of service accounts and role bindings increases. 

Currently, there is no straightforward way to determine which RoleBinding or ClusterRoleBinding is associated with a given serviceAccount using kubectl. This lack of visibility makes it difficult for cluster administrators to audit, debug, and manage access control effectively.

The proposed feature aims to address this gap by providing a simple command-line tool that cluster administrators can use to find related role bindings for a service account. This will enhance the manageability and security of Kubernetes clusters.

## 3. Detailed Design

The proposed feature will be implemented as a new kubectl subcommand, leveraging the existing Kubernetes Go libraries for API interactions. The command will be designed to take a service account name and namespace as input and query the Kubernetes API for related role bindings.

Here are the detailed steps:

- The command will be implemented in a new Go package for the kubectl plugin.
- The command will accept two required flags: `--name` for the service account name and `--namespace` for the service account namespace.
- The command will use the Kubernetes Go libraries to interact with the Kubernetes API, specifically to list RoleBindings and ClusterRoleBindings.
- For each RoleBinding and ClusterRoleBinding found, the command will check if the subject matches the given service account name and namespace.
- The command will output the names and types (RoleBinding or ClusterRoleBinding) of the related role bindings.

## 4. Drawbacks

The main drawbacks of this proposal are:

- Additional complexity: Introducing a new subcommand adds to the complexity of the kubectl tool.
- Performance impact: Querying the Kubernetes API for role bindings could potentially impact performance, especially in large clusters with many role bindings.

However, these drawbacks are considered minimal compared to the benefits of improved manageability and security.

## 5. Alternatives

An alternative approach could be to extend the existing `kubectl get` command to include role bindings when listing service accounts. However, this approach could lead to cluttered output and might not provide the flexibility and specificity that a dedicated subcommand can offer.

Another alternative is to implement this feature as a separate tool or script that interacts with the Kubernetes API. However, integrating this functionality directly into kubectl provides a more seamless and consistent user experience.
