# RFC: feat: Allow to add website URL for RAG in collections

### Summary

This RFC proposes the addition of a feature to allow users to add website URLs to collections for querying. The feature will enable users to query external websites, with optional authentication and scheduled updates.

### Motivation

The motivation behind this feature is to enhance the functionality of the RAG (Retrieve, Augment, Generate) system by allowing users to query external websites. This will enable users to retrieve information from external sources, augment it with internal knowledge, and generate more accurate and informative responses. The feature will also enable users to integrate with external tools like Confluence, making it easier to retrieve information from these sources.

### Detailed Design

The proposed design involves the following components:

*   **URL Collection System:** A new system will be designed to allow users to add multiple URLs to collections. Each URL will have its own settings, including authentication settings (e.g., username, password, API key).
*   **Authentication:** The system will support basic authentication (username and password) and API key authentication. Users will be able to add authentication settings for each URL.
*   **Scheduled Scanning:** The system will allow users to schedule updates for each URL. Users will be able to choose the frequency of updates (e.g., every hour, every day).
*   **Integration with External Tools:** The system will provide integration with external tools like Confluence through their APIs. Users will be able to add Confluence pages to collections and query them.
*   **Rescan Button:** A rescan button will be added to allow users to manually trigger an update for a URL.

The implementation will involve the following steps:

1.  Design the URL collection system and database schema.
2.  Implement authentication for protected URLs.
3.  Implement scheduled scanning using modern JavaScript's async/await and timers.
4.  Integrate with external tools like Confluence through their APIs.

### Drawbacks

The following drawbacks are identified:

*   **Complexity:** Implementing authentication for protected URLs and integration with external tools may add complexity to the system.
*   **Security:** Handling authentication settings and API keys may introduce security risks if not implemented properly.
*   **Performance:** Scheduled scanning may impact system performance if not implemented efficiently.

### Alternatives

The following alternatives are considered:

*   **Manual Updates:** Users can manually update the content of external websites by re-scanning the URL. However, this approach may be time-consuming and prone to errors.
*   **Third-Party Services:** Users can use third-party services to retrieve information from external websites. However, this approach may require additional setup and configuration.

### Unresolved Questions

The following questions remain unresolved:

*   **How will we handle rate limiting and IP blocking from external websites?**
*   **How will we ensure the security of authentication settings and API keys?**
*   **How will we optimize the performance of scheduled scanning?**
