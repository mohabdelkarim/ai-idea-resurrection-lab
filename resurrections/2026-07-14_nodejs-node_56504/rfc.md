# RFC: A workflow to periodically check status for author-ready PRs opened by a non-collaborator

## 1. Summary

This RFC proposes the creation of a GitHub Actions workflow to periodically check the status of author-ready PRs opened by non-collaborators in the nodejs/node repository. The workflow aims to ensure that these PRs do not stall and that contributors receive timely feedback on their submissions.

## 2. Motivation

The nodejs/node repository receives numerous PRs from contributors who are not collaborators. While these PRs are often labeled as 'author-ready' by the contributors, indicating they are ready for review, they sometimes stall without receiving the necessary attention from collaborators. This can lead to contributors waiting indefinitely for feedback, which may cause them to lose interest or seek opportunities elsewhere.

## 3. Detailed Design

The proposed workflow will be implemented using GitHub Actions and will operate as follows:

- **Trigger**: The workflow will trigger on a schedule (e.g., every 7 days).
- **Filter PRs**: It will fetch a list of all PRs labeled as 'author-ready' that were opened by non-collaborators.
- **Check Status**: For each PR, it will check the status (e.g., last comment, last update) to determine if the PR has stalled.
- **Notification**: If a PR is found to have stalled, the workflow will send a notification to the collaborators involved, suggesting they review and move the PR forward.

The workflow will be written in TypeScript, leveraging the GitHub Actions Toolkit for Node.js to interact with the GitHub API.

## 4. Drawbacks

- **Additional Load**: Implementing this workflow may add load to the repository's GitHub Actions runners, potentially affecting the performance of other workflows.
- **Notification Fatigue**: Collaborators might experience notification fatigue if they receive too many pings about stalled PRs, potentially leading to decreased responsiveness.

## 5. Alternatives

- **Manual Process**: Rely on manual checks by collaborators or maintainers to follow up on stalled PRs. This approach is prone to human error and is not scalable.
- **Existing Tools**: Utilize existing tools or bots that can perform similar tasks, such as stale-bots or similar GitHub Actions workflows. However, these might not offer the customization needed for this specific use case.

## 6. Unresolved Questions

- **Definition of 'Stalled'**: How to accurately define a PR as 'stalled'? This could be based on the lack of activity (comments, updates) over a certain period.
- **Notification Mechanism**: What is the best mechanism for notifying collaborators about stalled PRs? This could include adding a comment to the PR, sending an email, or pinging them directly on GitHub.
- **Opt-out Mechanism**: Should there be an opt-out mechanism for collaborators who do not wish to receive these notifications?
