# RFC: umd module compiler option doesn't have a fallback for global namespace.

## Summary

TypeScript's UMD (Universal Module Definition) module compiler option currently does not provide a fallback for the global namespace. This limitation causes issues for library developers who want to ensure their code works seamlessly in various environments, including browser globals and module systems. This RFC proposes adding a new compiler option to generate UMD modules with a fallback for the global namespace.

## Motivation

The UMD module system is widely used in JavaScript libraries to ensure compatibility across different environments, including browsers, Node.js, and module systems like AMD and CommonJS. However, the current TypeScript implementation of UMD modules does not provide a fallback for the global namespace, which can lead to issues when the library is used in a browser or another environment without a module system.

Library developers need to ensure their code works in various scenarios, including:

* Browser globals: When used directly in a browser, the library should expose its main entry point as a global variable.
* Module systems: When used with module systems like AMD, CommonJS, or ES modules, the library should work as expected.

The lack of a global namespace fallback in UMD modules hinders the development of compatible libraries.

## Detailed Design

To address this limitation, we propose adding a new compiler option, `umdGlobalFallback`, which defaults to `false` to preserve backwards compatibility. When `umdGlobalFallback` is set to `true`, the TypeScript compiler will generate UMD modules with a fallback for the global namespace.

The UMD pattern will be updated to include a check for the global object (usually `window` or `global`) and assign the library's main entry point to it. This ensures that when the library is used in a browser or another environment without a module system, it will still expose its main entry point as a global variable.

For example, given a simple library:

typescript
// mylib.ts
export function greet(name: string) {
  console.log(`Hello, ${name}!`);
}
The generated UMD module with `umdGlobalFallback` set to `true` would look like this:

javascript
// mylib.umd.js
(function (root, factory) {
  if (typeof define === 'function' && define.amd) {
    define([], factory);
  } else if (typeof exports === 'object') {
    module.exports = factory();
  } else {
    root.mylib = factory();
  }

  function greet(name) {
    console.log('Hello, ' + name + '!');
  }
  return {
    greet: greet
  };
}));
## Drawbacks

*   Backwards compatibility: The introduction of a new compiler option and changes to the UMD module generation may affect existing codebases that rely on the current behavior.
*   Potential issues with global namespace pollution: The fallback for the global namespace may lead to issues with global namespace pollution if not implemented carefully.

## Alternatives

*   Instead of adding a new compiler option, we could modify the existing UMD module generation to always include a fallback for the global namespace. However, this approach may break existing codebases that rely on the current behavior.
*   Another alternative is to provide a separate compiler option for generating a global namespace fallback, but this would add more complexity to the compiler options.

## Unresolved Questions

*   How to handle cases where the library has multiple entry points? Should we generate a separate UMD module for each entry point, or provide a way to configure the global namespace fallback for each entry point?
*   What are the implications of this feature on the TypeScript compiler's performance and memory usage?
