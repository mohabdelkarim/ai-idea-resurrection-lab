# RFC: Increase verbosity on a per task basis

### 1. Summary

This RFC proposes to enhance Ansible's verbosity control by introducing per-task verbosity settings. The goal is to allow users to increase verbosity on a per-task basis in Ansible playbooks, providing more granular control over logging and debugging.

### 2. Motivation

Currently, Ansible's verbosity is controlled globally, which can lead to overwhelming logs when debugging specific tasks. By introducing per-task verbosity, users can focus on debugging individual tasks without affecting the verbosity of the entire playbook. This feature will improve the overall usability and debuggability of Ansible.

### 3. Detailed Design

To implement per-task verbosity, we will introduce a new task keyword, 'verbosity', which accepts an integer value. This value will override the global verbosity setting for the specific task. We will utilize Ansible's existing logging mechanisms and modify the task executor to handle the new verbosity setting.

The feature will be implemented as a plugin to ensure flexibility and backward compatibility. The plugin will be designed to work with Ansible's existing logging system, allowing users to configure logging levels and verbosity on a per-task basis.

The proposed implementation includes the following steps:

* Introduce a new task keyword, 'verbosity', which accepts an integer value.
* Modify the task executor to handle the new verbosity setting.
* Integrate the feature with Ansible's existing logging mechanisms.

### 4. Drawbacks

The main drawbacks of this feature are:

* Potential performance impact due to increased logging.
* Additional complexity in the task executor and logging mechanisms.

However, these drawbacks can be mitigated by:

* Implementing the feature as a plugin to minimize the impact on the core Ansible codebase.
* Providing users with control over logging levels and verbosity to minimize performance overhead.

### 5. Alternatives

Alternative solutions considered include:

* Using a global verbosity setting with a task-level override.
* Implementing a separate logging system for tasks.

However, these alternatives were rejected due to:

* Limited flexibility and granularity.
* Increased complexity and potential performance impact.

### 6. Unresolved Questions

The following questions remain unresolved:

* How to handle task-level verbosity inheritance (e.g., do tasks inherit verbosity from the parent play or block?).
* How to interact with existing logging mechanisms and plugins.

These questions will be addressed during the implementation phase, and the solution will be refined based on community feedback and testing.
