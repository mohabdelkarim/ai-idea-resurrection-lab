# RFC: max-retries-exceeded exceptions are confusing

Summary

--------

This RFC proposes improvements to the handling of max-retries-exceeded exceptions in the `requests` library. The goal is to make these exceptions more informative and user-friendly, allowing developers to better understand and address retry failures.



Motivation

-----------

The current implementation of max-retries-exceeded exceptions in `requests` can be confusing and lacks clear guidance on what went wrong and how to fix it. This is largely due to the limitations of the underlying `urllib3` library's retry mechanism and the lack of customization options available to users. With the evolution of modern Python libraries and practices around error handling and retries, it is now feasible to introduce a more user-friendly and informative approach.



Detailed Design

----------------

To address this issue, we propose the following changes:

1. Introduction of a new exception type, `requests.exceptions.MaxRetriesExceededError`, which will provide clearer guidance on what went wrong and potentially how to fix it.

2. Update the `requests` library to use a more configurable retry mechanism, such as the `tenacity` library, allowing users to specify their retry policies more explicitly.

3. Modify the `ConnectionError` handling in `requests/adapters.py` to catch and re-raise the exception with more context, including information about the failed request and the retry policy that was applied.



Drawbacks

----------

The main drawbacks of this proposal are:

1. Backward compatibility: Changing the existing exception type and behavior may break existing code that relies on the current behavior.

2. Added complexity: Introducing a new exception type and a more configurable retry mechanism may add complexity to the library, potentially making it harder to maintain and understand.



Alternatives

------------

Alternative solutions considered include:

1. Improving the existing exception message to provide more context, without introducing a new exception type.

2. Using a different library or approach for handling retries, such as `backoff` or `retrying`.

However, these alternatives were deemed less desirable due to their limitations in addressing the root causes of the issue.



Unresolved Questions

--------------------

The following questions remain to be addressed:

1. How to balance backward compatibility with the need for a more informative and user-friendly exception handling approach?

2. What is the optimal way to integrate the `tenacity` library or a similar retry mechanism into the `requests` library?

3. How to ensure that the new exception type and behavior are properly documented and communicated to users?
