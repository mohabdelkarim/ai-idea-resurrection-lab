# RFC: .includes or .indexOf does not narrow the type

## 1. Summary

This RFC proposes to enhance TypeScript's type narrowing capabilities when using methods like `.includes()` or `.indexOf()` on union types. Currently, TypeScript does not narrow the type of a union type when these methods return a boolean value indicating the presence of a value. We aim to improve this by introducing a type guard that leverages type predicates, allowing for more precise type narrowing.

## 2. Motivation

The current limitation in TypeScript can lead to cumbersome and less type-safe code when working with union types and checking for the presence of a value. For instance:

typescript
let message: string | number;
if (['hello', 'world'].includes(message)) {
  // Type of message is still string | number here
  console.log(message.toUpperCase()); // Error: Object is possibly 'number'.
}
By introducing a type guard that narrows the type based on the result of `.includes()` or `.indexOf()`, developers can write more expressive and type-safe code:

typescript
if (isStringInArray(['hello', 'world'], message)) {
  // Type of message is narrowed to string here
  console.log(message.toUpperCase()); // No error
}
## 3. Detailed Design

We propose introducing a type guard function, `isTInArray<T>(arr: T[], value: string | T): value is T`, which utilizes type predicates to narrow the type of `value` based on its presence in `arr`. This function can be used with both `.includes()` and `.indexOf()` to achieve type narrowing.

typescript
function isTInArray<T>(arr: T[], value: string | T): value is T {
  return arr.includes(value as T);
}

// Usage
let message: string | number;
if (isTInArray(['hello', 'world'], message)) {
  // Type of message is narrowed to string
  console.log(message.toUpperCase()); // No error
}
For arrays of strings, we can create a specialized type guard:

function isStringInArray(arr: string[], value: string | number): value is string {
  return arr.includes(value as string);
}

// Usage

## 4. Drawbacks

One potential drawback is the additional complexity introduced by the need for a type guard function. However, this can be mitigated by making the type guard functions part of a standard library or by providing a simple way to generate these functions through a tool or a code snippet.

## 5. Alternatives

An alternative approach could involve enhancing the TypeScript compiler to automatically narrow types based on `.includes()` or `.indexOf()` results. However, this might introduce significant complexity into the compiler and could have performance implications.
