# RFC: Feature Request: Idle model unload timeout (router mode / config.ini)

Summary

--------

This RFC proposes the implementation of an idle model unload timeout feature in llama-server. The feature will allow the server to automatically unload models that have been idle for a specified period of time, freeing up resources and improving overall efficiency.


Motivation

-----------

The motivation behind this feature is to optimize resource utilization in llama-server. Large language models require significant computational resources, and unloading idle models can help reduce memory usage and improve system responsiveness. This feature is particularly relevant in scenarios where multiple models are deployed and need to be managed efficiently.


Detailed Design

----------------

The idle model unload timeout feature will be implemented as follows:

1. A new configuration option 'unload-idle-seconds' will be introduced, which specifies the number of seconds after which an idle model will be unloaded. This option will default to 300 seconds.

2. The 'unload-idle-seconds' option will be configurable via the config.ini file or as a command-line argument.

3. The llama-server will track the idle time of each model and unload it when the specified time is exceeded.

4. The existing model management logic will be modified to include a timer for each model, which will be updated whenever the model is used.

5. When a model's idle time exceeds the specified threshold, the server will automatically unload the model.


Drawbacks

----------

The following drawbacks are anticipated:

1. Potential performance impact: Unloading and reloading models can incur a performance penalty, particularly if the model is large or complex.

2. Increased complexity: The feature will add complexity to the model management logic, which may lead to bugs or issues.

3. Potential for model thrashing: If the unload threshold is set too low, models may be repeatedly unloaded and reloaded, leading to thrashing and decreased performance.


Alternatives

------------

The following alternatives were considered:

1. Implement a simple model caching mechanism, where models are cached for a fixed period of time before being unloaded.

2. Introduce a more sophisticated model management system, which takes into account factors such as model priority, usage patterns, and system resources.

However, the proposed solution is considered the most straightforward and effective way to address the issue.


Unresolved Questions

--------------------

The following questions remain unresolved:

1. What is the optimal default value for the 'unload-idle-seconds' option?

2. How should the feature interact with other model management mechanisms, such as model prioritization or caching?

3. What are the performance implications of unloading and reloading models, and how can they be mitigated?

These questions will need to be addressed through further testing and evaluation of the feature.
