# RFC: Feature Request: Option to not check included files in clang-tidy

# Summary

This RFC proposes the addition of a new option to `clang-tidy`, allowing users to skip analyzing included files during the linting process. The goal is to improve the performance of `clang-tidy` by reducing the number of files that need to be analyzed, particularly in large projects with many included files.




# Motivation

`clang-tidy` is a powerful tool for identifying and fixing common issues in C and C++ code. However, its analysis can be slow, especially for large projects with many included files. Currently, `clang-tidy` analyzes all files, including included ones, which can lead to redundant checks and decreased performance. By introducing an option to skip analyzing included files, users can significantly speed up the linting process, making it more practical for large-scale projects.




# Detailed Design

The proposed feature will be implemented by adding a new flag to `clang-tidy`, `--skip-included-files`. When this flag is enabled, `clang-tidy` will skip analyzing included files. The implementation will involve modifying the `clang-tidy` driver to handle this new flag and adjust its analysis accordingly.



Key steps in the implementation:

1. **Introduce the new flag**: Add a new command-line flag `--skip-included-files` to the `clang-tidy` driver.
2. **Modify the analysis pipeline**: Update the analysis pipeline to check for the presence of the `--skip-included-files` flag. If the flag is set, skip analyzing included files.
3. **Utilize existing infrastructure**: Leverage the existing Clang tooling infrastructure to implement this feature, ensuring minimal disruption to the current codebase.
4. **Plugin pattern**: Consider implementing the feature using the plugin pattern, allowing for easy extension and customization.




# Drawbacks

1. **Potential loss of coverage**: Skipping analysis of included files might lead to missing some issues that are present in those files.
2. **Complexity in flag handling**: Adding a new flag requires careful handling to ensure it works correctly with existing flags and configurations.




# Alternatives

1. **Configuration-based approach**: Instead of a command-line flag, consider implementing this feature through a configuration option. This would allow users to set the behavior persistently through a configuration file.
2. **Per-file analysis control**: Another approach could be to provide a way to control analysis on a per-file basis, allowing users to selectively skip analysis of specific files or directories.
