# RFC: docker-compose copy file or directory to container

# Summary

This RFC proposes the implementation of a new command in `docker-compose` to copy files or directories to a container. The command, `docker-compose cp`, will leverage the Docker API to perform the copy operation, enhancing the usability and flexibility of `docker-compose` for managing containerized applications.

## Motivation

The ability to copy files or directories to a running container is a common requirement for developers and operators working with containerized applications. Currently, users have to use the `docker cp` command directly, which involves finding and specifying the container ID or name. Integrating this functionality into `docker-compose` will streamline workflows and improve user experience by allowing users to manage file transfers within the context of their Compose projects.

## Detailed Design

The proposed feature will be implemented as a new command in the `docker-compose` CLI, named `cp`. This command will accept the following arguments:

- `SOURCE`: The source file or directory to copy from the host.
- `DESTINATION`: The destination path inside the container.
- `SERVICE`: The name of the service (as defined in the Compose file) to which the container belongs.

The command will use the Docker API to interact with the container's filesystem. Specifically, it will utilize the `docker cp` endpoint, which allows for copying files and directories to and from a container.

The implementation will involve:
1. Defining a new struct, `CopyCommand`, which will handle the copy operation.
2. Integrating the `CopyCommand` into the existing `docker-compose` command-line interface.
3. Parsing the command arguments and validating their correctness.
4. Using the Docker Go client to perform the file copy operation.

Example usage:
docker-compose cp ./source:/path/in/container myservice
## Drawbacks

1. **Additional Complexity**: Adding a new command increases the complexity of the `docker-compose` codebase.
2. **Overlap with Docker CLI**: The `docker cp` command already provides this functionality. However, integrating it into `docker-compose` offers a more streamlined experience for users managing Compose projects.

## Alternatives

1. **Using `docker cp` Directly**: Users can continue using the `docker cp` command directly with the container ID or name. However, this requires additional steps to identify the container and does not integrate well with Compose project management.
2. **Custom Scripts**: Users could write custom scripts to achieve this functionality. However, this approach lacks the integration and polish of a native `docker-compose` command.

## Unresolved Questions

1. **Handling Service Names**: How to handle cases where a service name is not provided or is ambiguous? 
   - Potential solution: Require the service name to be specified and validate it against the Compose file.
2. **Path Resolution**: How to resolve relative paths for the source and destination?
   - Potential solution: Use the current working directory for relative paths and provide clear documentation on path resolution.
