# RFC: Add a (singular) `$prop` rune

## Summary

This RFC proposes the introduction of a singular `$prop` rune in Svelte, allowing for concise prop declarations with types. The goal is to improve the developer experience by reducing boilerplate code and making prop declarations more straightforward.

## Motivation

Currently, in Svelte, declaring props with types can be verbose and cumbersome. The existing `$props` rune requires developers to declare all props at once, which can lead to cluttered code and make it harder to manage individual props. The proposed `$prop` rune would enable developers to declare props one by one, with their respective types, making the code more readable and maintainable.

## Detailed Design

The `$prop` rune would be used to declare a single prop with a type. The syntax would be as follows:

svelte
<script>
  let someProp = $prop<string>('default');
</script>
This would declare a prop `someProp` of type `string` with a default value of `'default'`. The equivalent code using the existing `$props` rune would be:

svelte
<script>
  let { someProp = 'default' } = $props<{ someProp: string }>();
</script>
The `$prop` rune would also support more complex type declarations, such as:

svelte
<script>
  let someProp: { foo: string; bar: number } = $prop<{ foo: string; bar: number }>({ foo: 'baz', bar: 42 });
</script>
## Drawbacks

One potential drawback of introducing the `$prop` rune is that it may add complexity to the Svelte compiler and runtime. Additionally, it may take some time for developers to get used to the new syntax, which could lead to a short-term decrease in productivity.

## Alternatives

One alternative to the `$prop` rune is to extend the existing `$props` rune to support more concise syntax for declaring individual props. However, this approach may lead to inconsistencies in the API and make it harder for developers to understand the differences between `$props` and `$prop`.

Another alternative is to rely on TypeScript's built-in type inference and not introduce a new rune at all. However, this approach may not provide the same level of conciseness and readability as the `$prop` rune.
