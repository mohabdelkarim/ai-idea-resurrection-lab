# RFC: Suggestion: Range as Number type

# Summary

This RFC proposes the introduction of a range type in TypeScript, allowing developers to define a range of numbers as a type. The proposed type operator, `Range<T, U>`, would enable the creation of types that represent a range of numbers, enhancing the expressiveness and flexibility of TypeScript's type system.

## Motivation

The motivation behind this proposal is to provide a more robust and expressive way to define numeric ranges in TypeScript. Currently, developers have to use workarounds such as enums or literal types to achieve similar results, which can be cumbersome and limited. A range type would allow for more precise type definitions, better code readability, and improved maintainability.

## Detailed Design

The proposed design involves introducing a new type operator, `Range<T, U>`, where `T` and `U` are numeric types. This operator would create a new type that represents a range of numbers from `T` to `U` (inclusive). The type would be implemented using a combination of type inference and type checking.

For example:
type TTerminalColors = Range<0, 15>;
This would create a type `TTerminalColors` that represents a range of numbers from 0 to 15.

The range type would be subject to the following constraints:

* `T` and `U` must be numeric types (e.g., `number`, `bigint`, or a literal type that resolves to a number).
* `T` must be less than or equal to `U`.

The range type would also integrate with existing type guards and type checking mechanisms, allowing for more precise type checking and better code completion.

## Drawbacks

The main drawbacks of this proposal are:

* Adding a new type operator to the language, which could increase complexity and potentially lead to confusion.
* Potential performance implications due to the additional type checking and inference required.

However, these drawbacks are mitigated by the benefits of a more expressive and flexible type system, which would lead to better code quality and maintainability.
