# RFC: Feature Request: Importing a `js` file could fall back to `ts` file in a `ts` file

## Summary

This RFC proposes implementing a fallback import resolution strategy in Deno for TypeScript files to import '.js' files that may also have '.ts' counterparts. The goal is to enhance developer experience by allowing seamless imports of JavaScript modules from TypeScript files, with an optional fallback to TypeScript implementations if available.

## Motivation

Currently, when a Deno module written in TypeScript attempts to import a JavaScript file, it strictly adheres to the specified import path. If the imported file has both '.js' and '.ts' versions, but only the '.js' version is directly referenced in the import statement, Deno does not automatically resolve to the '.ts' version even if it exists. This can lead to a less-than-ideal developer experience, especially in mixed-language projects or when gradually migrating JavaScript projects to TypeScript.

The motivation behind this feature is to provide a more flexible and intelligent import resolution mechanism. By allowing TypeScript files to fall back to importing '.ts' files when '.js' files are referenced, Deno can better support projects that use both JavaScript and TypeScript, facilitating smoother transitions and interoperability between the two.

## Detailed Design

The proposed feature will be implemented through a new ImportMap resolver plugin in Deno. This plugin will introduce a fallback strategy for import resolutions:

1. **Import Statement Analysis**: When a TypeScript file attempts to import a module via a '.js' path, the plugin will intercept this import statement.
2. **Existence Check**: The plugin will first verify if a '.js' file matching the import path exists. If it does, the import proceeds as usual.
3. **Fallback to '.ts'**: If the '.js' file does not exist but a '.ts' file does, the plugin will automatically resolve the import to the '.ts' file instead. This ensures that TypeScript files can seamlessly import modules that have TypeScript implementations available.

To implement this feature, we will leverage Deno's plugin architecture, specifically the ` deno plugins` API, to create a custom ImportMap resolver. This resolver will utilize a strategy pattern to manage the fallback logic between '.js' and '.ts' files.

## Drawbacks

1. **Potential Ambiguity**: One potential drawback is the introduction of ambiguity in import paths. If both '.js' and '.ts' versions of a module exist, it's unclear which version should be prioritized. However, by strictly following a fallback strategy that prioritizes '.js' then '.ts', we minimize this ambiguity.
2. **Performance Impact**: Another concern is the potential performance impact of introducing an additional existence check for '.ts' files. This can be mitigated by caching the results of existence checks to avoid repeated file system operations.

## Alternatives

An alternative approach could involve explicitly declaring import priorities in a configuration file or import map. However, this would add complexity and require more effort from developers to maintain these configurations. The proposed solution aims to provide a sensible default that works out of the box with minimal configuration.

## Unresolved Questions

1. **Configurability**: Should there be an option to configure the fallback behavior, allowing developers to opt-out or modify the priority of '.js' vs '.ts' files?
2. **Interaction with Existing Plugins**: How will this feature interact with existing import resolution plugins or custom import maps? Will it be possible to combine this feature with other plugins that manipulate import paths?
3. **Type Safety**: How will Deno ensure type safety when falling back to '.ts' files for imports originally specified as '.js'? Will additional type checking be required to ensure compatibility?

Addressing these questions will be crucial to the successful implementation and adoption of this feature.
