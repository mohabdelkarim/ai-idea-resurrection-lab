# Summary

This RFC proposes the addition of experimental support for server-side rendering (SSR) with dynamic routes in Astro. The goal is to provide a flexible and efficient way to render dynamic pages on the server, while maintaining Astro's existing static site generation capabilities.




# Motivation

Astro's initial focus on launching as a static site builder led to the omission of server-side rendering (SSR) support. However, with the maturity of modern web development frameworks and growing demand for dynamic SSR, it's now feasible to reconsider this feature. The introduction of new tools and technologies, such as improved TypeScript support and enhanced Node.js performance, makes SSR more feasible. The success of similar features in other frameworks like Next.js and SvelteKit provides a clear path forward.




# Detailed Design

To implement SSR in Astro, we propose introducing a new adapter system, similar to SvelteKit, which allows for easy deployment to various platforms including Node.js and Cloudflare Workers. Dynamic pages can utilize file-based routing, borrowing concepts from Next.js. The implementation should include a mechanism for registering dynamic props, potentially through a `getServerSideProps` method. This feature can be initially released behind an experimental flag to ensure a smooth integration with Astro's existing static site generation capabilities.

The adapter system will provide a layer of abstraction between Astro and the deployment platform, allowing for easy integration with different environments. The file-based routing system will enable developers to create dynamic pages using a familiar and intuitive API. The `getServerSideProps` method will allow developers to fetch data on the server and pass it to the page component.

One potential drawback of adding SSR support is the increased complexity of the Astro codebase. This may lead to longer maintenance and development times. Additionally, SSR can introduce new performance considerations, such as handling server-side rendering of dynamic pages and managing cache invalidation.

Another potential drawback is the potential for conflicts with Astro's existing static site generation capabilities. To mitigate this, the SSR feature can be initially released behind an experimental flag, allowing developers to opt-in to the new feature and providing a clear path for migration.
