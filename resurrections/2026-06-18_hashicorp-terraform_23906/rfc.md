# RFC: Request: Enable simple/native way to read .env files

### Summary

This RFC proposes the implementation of a native, simple way to read .env files in Terraform configurations. The goal is to enhance usability by allowing users to easily load environment variables from .env files without requiring external tools or complex workarounds.

### Motivation

Currently, Terraform users who want to utilize .env files for environment variable management have to rely on external tools or manual variable assignments. This can lead to cumbersome workflows and potential configuration errors. Native support for reading .env files would streamline the process of managing environment variables directly within Terraform configurations.

The motivation behind this proposal is to:

- Simplify the management of environment variables.
- Enhance user experience by integrating a commonly requested feature.
- Maintain consistency with Terraform's configuration-as-code philosophy.

### Detailed Design

To implement this feature, Terraform will introduce a new data source, `data.env_file`, specifically designed for reading .env files. This data source will be configured with a path to the .env file and optionally, a list of variables to extract.

The proposed implementation details are as follows:

- **Data Source Configuration**: The `data.env_file` source will accept a required `path` attribute pointing to the .env file and an optional `variables` attribute, which is a list of strings representing the variable names to extract from the .env file.

- **.env File Parsing**: Terraform will use a reliable library (e.g., dotenv for Go) to parse the .env file. This library will handle the parsing logic, including comments, empty lines, and variable assignments.

- **Variable Exposure**: The parsed variables will be exposed as attributes of the `data.env_file` resource. Users can then reference these variables in their Terraform configurations.

- **Example Configuration**:

data "env_file" "example" {
  path      = "${path.module}/.env"
  variables = ["DB_HOST", "DB_PORT"]
}

resource "example_resource" "example" {
  host = data.env_file.example.variables.DB_HOST
  port = data.env_file.example.variables.DB_PORT
}
### Drawbacks

- **Security Considerations**: Loading variables from .env files could potentially expose sensitive information if not handled properly. It's crucial to ensure that .env files are not committed to version control and are properly ignored by .gitignore or similar mechanisms.

- **Configuration Management**: Introducing a new data source adds to the complexity of Terraform's configuration model. However, this complexity is manageable and aligns with Terraform's extensibility goals.

### Alternatives

- **External Tools**: Users currently employ external tools (e.g., shell scripts, dotenv) to load .env files before running Terraform. While effective, this approach requires additional steps and tooling.

- **terraform.tfvars**: Terraform supports loading variables from `terraform.tfvars` files. However, these files are not designed for environment variable management in the same way .env files are.

### Unresolved Questions

- **Error Handling**: How should Terraform handle parsing errors in .env files? Should it fail loudly or attempt to continue with a partial set of variables?

- **Variable Validation**: Should Terraform validate the variables extracted from .env files against a predefined schema or type?

- **Integration with Existing Features**: How will this feature interact with existing Terraform features like sensitive variables and variable validation?

By addressing these questions and implementing the proposed solution, Terraform can provide a native, user-friendly way to integrate .env files into configurations, enhancing the overall usability and flexibility of the tool.
