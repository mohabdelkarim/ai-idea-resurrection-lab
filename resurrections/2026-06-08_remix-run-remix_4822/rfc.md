# RFC: hydration error when there is a chrome extension that modifies DOM

Summary
--------

Remix hydration fails when Chrome extensions modify the DOM, causing style issues. This RFC proposes a solution to modify Remix's hydration process to account for external DOM modifications, ensuring a seamless user experience.

Motivation
------------

Chrome extensions can modify the DOM, causing Remix's hydration process to fail. This results in style issues and a broken user interface. The current implementation does not handle such edge cases, leading to unpredictable behavior. With the growing adoption of Remix and the increasing use of Chrome extensions, it's essential to address this issue.

Detailed Design
---------------

To solve this problem, we propose the following:

1.  **DOM Validation**: Before hydration, Remix will validate the DOM to detect any modifications made by external scripts, including Chrome extensions.
2.  **Customizable Hydration**: Introduce a new API or configuration option that allows developers to customize the hydration process. This can include providing a custom DOM validation function or specifying how to handle modifications.
3.  **Error Handling**: Leverage Remix's built-in error handling mechanisms to provide a better user experience. If a modification is detected, Remix can display an error message or fallback to a safe mode.

The proposed solution involves modifying the `hydrate` function in Remix to include the following steps:

*   Check if the DOM has been modified by an external script.
*   If modifications are detected, use the customizable hydration option to handle them.
*   If no custom handling is provided, display an error message or fallback to a safe mode.

Drawbacks
----------

The proposed solution has the following drawbacks:

*   **Performance Overhead**: The additional DOM validation step may introduce a performance overhead.
*   **Complexity**: Customizable hydration and error handling may add complexity to the Remix API.

However, these drawbacks can be mitigated by:

*   Optimizing the DOM validation step to minimize performance impact.
*   Providing clear documentation and examples for the customizable hydration API.

Alternatives
------------

The following alternatives were considered:

*   **Ignore Modifications**: Ignore DOM modifications made by external scripts and continue with hydration. However, this approach may lead to unpredictable behavior and style issues.
*   **Disable Hydration**: Disable hydration when external modifications are detected. However, this approach may impact the user experience and prevent Remix from functioning correctly.

Unresolved Questions
--------------------

The following questions remain unresolved:

*   **How to handle modifications made by multiple extensions?**
*   **What is the best approach to validate the DOM without introducing significant performance overhead?**
*   **How to provide a seamless user experience when modifications are detected?**

These questions will be addressed through further discussion and testing. The proposed solution provides a foundation for handling DOM modifications made by Chrome extensions, and future iterations can refine and improve the approach.
