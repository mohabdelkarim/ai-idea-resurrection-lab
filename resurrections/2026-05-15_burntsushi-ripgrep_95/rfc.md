# RFC: fulltext search

Summary

--------

This RFC proposes the integration of fulltext search capabilities into ripgrep, a popular tool for searching code. The goal is to enhance the efficiency and effectiveness of code search by leveraging modern search libraries and techniques.


Motivation

-----------

ripgrep is widely used for searching code due to its speed and simplicity. However, it currently lacks fulltext search capabilities, which are essential for efficiently locating specific code snippets within large codebases. The integration of fulltext search into ripgrep would significantly improve the user experience by providing more accurate and relevant search results.


Detailed Design

----------------

The proposed design involves the following components:

1. **FulltextSearcher Module**: A new module will be added to the ripgrep core, responsible for handling indexing and querying.

2. **Indexing Library**: The `tantivy` library will be used for indexing and search operations.

3. **Serialization Library**: The `bincode` library will be used for efficient serialization of index data.

4. **Interaction with ripgrep Parser**: The `FulltextSearcher` module will interact with the ripgrep parser to extract relevant code information for indexing.

5. **CQRS Pattern**: The design will follow the Command-Query Responsibility Segregation (CQRS) pattern to separate indexing and search logic.



The workflow will be as follows:

- The user initiates an indexing operation on a codebase.

- The `FulltextSearcher` module uses the ripgrep parser to extract code information.

- The extracted information is indexed using `tantivy`.

- The indexed data is serialized using `bincode` for storage.

- When a search query is issued, the `FulltextSearcher` module queries the index and returns relevant results.


Drawbacks

----------

1. **Additional Complexity**: Integrating fulltext search will add complexity to ripgrep, potentially affecting its performance and maintainability.

2. **Index Maintenance**: The index will need to be updated periodically to reflect changes in the codebase, which could be resource-intensive.

3. **User Interaction**: The user interface for managing indexes and issuing search queries will need to be designed and implemented.


Alternatives

------------

1. **External Tools**: Users could use external tools like Elasticsearch or SQLite FTS for fulltext search, but this would require additional setup and integration.

2. **Existing ripgrep Features**: Users could continue using ripgrep's existing features, such as regex searches, but these may not provide the same level of accuracy and efficiency as fulltext search.


Unresolved Questions

---------------------

1. **Indexing Strategy**: What is the optimal strategy for indexing codebases, particularly large ones?

2. **Query Language**: What query language should be supported for search queries?

3. **Performance Metrics**: What performance metrics should be used to evaluate the effectiveness of the fulltext search integration?
