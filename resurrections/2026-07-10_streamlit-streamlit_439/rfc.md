# RFC: Support custom HTTP requests

## 1. Summary

This RFC proposes the addition of custom HTTP request support to Streamlit, enabling developers to extend the functionality of their apps by handling arbitrary HTTP requests.

## 2. Motivation

Streamlit's simplicity and ease of use have made it a popular choice for building data applications. However, as apps grow in complexity, the need for custom HTTP requests arises. Supporting custom HTTP requests will allow developers to:

* Integrate with external services and APIs
* Build RESTful APIs alongside their Streamlit apps
* Enhance app functionality with custom endpoints

## 3. Detailed Design

To implement custom HTTP requests in Streamlit, we propose the following:

* Introduce a new module, `streamlit.server`, to manage custom routes and requests
* Leverage Python's async capabilities and libraries like aiohttp or httpx to handle HTTP requests
* Provide a simple API for defining custom routes and handlers, such as `@st.server.route('/api/data')`
* Integrate with Streamlit's existing app lifecycle to ensure seamless execution

The implementation will consider:

* Backward compatibility with existing Streamlit apps
* Potential performance implications and optimize accordingly

## 4. Drawbacks

The proposed feature may introduce:

* Additional complexity to the Streamlit codebase
* Potential security risks if not implemented correctly
* Performance overhead due to the handling of custom requests

## 5. Alternatives

Alternative approaches considered:

* Using existing libraries like Flask or FastAPI alongside Streamlit
* Implementing a custom solution using Python's built-in HTTP libraries

However, integrating custom HTTP request support directly into Streamlit provides a more streamlined and user-friendly experience.

## 6. Unresolved Questions

* How to handle authentication and authorization for custom routes?
* What is the optimal way to handle errors and exceptions for custom requests?
* How to ensure the feature does not introduce significant performance overhead?

These questions will be addressed during the implementation phase, and the community will be engaged to gather feedback and suggestions.
