# RFC: Support for multiple ports in annotations

Summary

    --------

    This RFC proposes an enhancement to Prometheus to support multiple ports in annotations, allowing for more flexible service discovery and scraping of metrics from various ports of a service.

    

    Motivation

    ---------

    Currently, Prometheus annotations support specifying a single port for scraping metrics. However, with the increasing complexity of modern applications and the use of container orchestration systems like Kubernetes, there is a growing need to scrape metrics from multiple ports of a service. This could be due to exposing different types of metrics (e.g., HTTP, gRPC, or custom protocols) on separate ports or serving multiple applications within the same pod or container, each exposing metrics on different ports.

    

    Detailed Design

    --------------

    To implement support for multiple ports in annotations, we propose the following:

    1. **Annotation Format**: Extend the existing annotation format to support specifying multiple ports. This could be achieved by using a comma-separated list of ports (e.g., `prometheus.io/scrape: 'true', prometheus.io/port: '8080,8081'`) or a more structured format like JSON (e.g., `prometheus.io/scrape: 'true', prometheus.io/ports: '{"http": 8080, "grpc": 8081}'`).

    2. **Configuration Option**: Introduce a new configuration option in the Prometheus server to enable or disable the parsing of multiple ports from annotations. This ensures backwards compatibility and allows users to opt-in to this feature.

    3. **Service Discovery Updates**: Update the Kubernetes service discovery component to properly handle and expose multiple ports from annotations. This may involve changes to how targets are created and how their ports are determined.

    4. **Scraping Mechanism Adjustments**: Adjust the scraping mechanism to handle multiple ports per target. This could involve creating separate scrape targets for each port or modifying the existing scrape target to handle multiple ports.

    

    Drawbacks

    ---------

    1. **Complexity**: Implementing support for multiple ports in annotations adds complexity to the Prometheus service discovery and scraping mechanisms.

    2. **Backwards Compatibility**: There is a risk of impacting existing users who rely on the current single-port annotation behavior.

    3. **Configuration and Management**: Supporting multiple ports per target may complicate the configuration and management of Prometheus and related tools.

    

    Alternatives

    ------------

    1. **Use of Service Discovery Mechanisms**: Instead of relying on annotations, users could utilize existing service discovery mechanisms (e.g., Kubernetes service discovery) to create separate targets for each port.

    2. **Sidecar Containers**: Users could deploy sidecar containers to handle metrics collection for different ports, forwarding metrics to a single port scraped by Prometheus.

    

    Unresolved Questions

    -------------------

    1. **Optimal Annotation Format**: What is the most efficient and user-friendly annotation format for specifying multiple ports?

    2. **Default Behavior**: Should the support for multiple ports in annotations be enabled by default or require explicit configuration?

    3. **Impact on Performance**: What is the potential impact on Prometheus performance when handling multiple ports per target, and how can it be mitigated?
