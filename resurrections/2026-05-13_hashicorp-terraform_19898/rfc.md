# RFC: Optional arguments in object variable type definition

Summary

    --------

    This RFC proposes adding support for optional arguments in object variable type definitions in Terraform. The goal is to enhance the flexibility and robustness of Terraform's type system, allowing for more precise and user-friendly variable definitions.

    

    Motivation

    ---------

    Currently, Terraform's object type definitions require all properties to be specified. However, in many cases, some properties are optional or have default values. The lack of support for optional arguments leads to verbose configurations and makes it difficult to manage complex infrastructures. This feature will align Terraform with other configuration languages and improve the overall user experience.

    

    Detailed Design

    --------------

    To implement this feature, we propose introducing an `optional` keyword for object properties. This keyword will allow users to define properties that can be omitted when creating an object. The type of an optional property will be the same as its required counterpart, but it will be marked as optional.

    

    Example:

    ```hcl

    variable "example" {

      type = object({

        required_property = string

        optional_property = optional(string)

      })

    }

    ```

    In this example, `required_property` must always be provided, while `optional_property` can be omitted.

    

    The implementation will involve updating the `ObjectType` and `Property` classes in Terraform's configuration parser to accommodate optional properties. We will also need to enhance type checking and validation to ensure that optional properties are handled correctly.

    

    Drawbacks

    ---------

    One potential drawback of this feature is that it may introduce additional complexity to Terraform's type system. However, we believe that the benefits of improved flexibility and usability outweigh the costs. Additionally, the implementation will be designed to be backward compatible, ensuring that existing configurations continue to work as expected.

    

    Alternatives

    ------------

    One alternative approach is to use a `default` value for optional properties. However, this approach has limitations, as it does not provide a clear indication of which properties are optional. Another alternative is to use a separate `optional` object type, but this would lead to a more complex type system and is not necessary given the proposed solution.
