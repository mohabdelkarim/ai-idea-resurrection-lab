# RFC: ActiveRecord enum: use validation if exists instead of raising ArgumentError

# Summary

This RFC proposes a change to the behavior of ActiveRecord's enum feature. Currently, when an invalid value is assigned to an enum attribute, an ArgumentError is raised. This RFC suggests that instead of raising an error, the enum should use validation if it exists for the attribute.




# Motivation

The current behavior of raising an ArgumentError can be restrictive and does not align with the more flexible validation mechanisms provided by Rails. By using validation if it exists, developers can handle invalid enum values in a more controlled and application-specific way. This change will make ActiveRecord enums more adaptable to different application requirements and error handling strategies.




# Detailed Design

The proposed change involves modifying the setter method generated by ActiveRecord for enum attributes. When setting an enum value, the code should check if a validation exists for the attribute. If a validation exists, it should use that validation to check the value; otherwise, it falls back to raising an ArgumentError.

Here's a high-level example of how this could be implemented:

ruby
class EnumValidatorExample < ApplicationRecord
  enum status: [:active, :inactive]

  validates :status, presence: true
end

# Usage
example = EnumValidatorExample.new
begin
  example.status = 'unknown' # This should validate and not raise an error
rescue ArgumentError => e
  puts e.message
end
In this example, assigning 'unknown' to the status attribute should trigger the validation check instead of immediately raising an ArgumentError.




# Drawbacks

1. **Backward Compatibility**: This change could potentially break existing applications that rely on the ArgumentError being raised for invalid enum values. Developers would need to update their code to handle validation errors instead.

2. **Performance**: Adding an additional validation check could introduce a minor performance hit, although this is expected to be negligible in most cases.

3. **Complexity**: Integrating this change with existing error handling mechanisms could add complexity, particularly for applications with custom error handling.




# Alternatives

1. **Leave the Current Behavior**: Maintain the status quo and continue to raise an ArgumentError for invalid enum values. This approach ensures consistency but does not provide the flexibility that validation offers.

2. **Introduce a Global Configuration Option**: Provide a global configuration option to choose between raising an error or using validation for invalid enum values. This approach allows developers to opt-in or opt-out of the new behavior.
