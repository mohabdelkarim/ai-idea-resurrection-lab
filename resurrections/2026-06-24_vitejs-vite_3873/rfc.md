# RFC: vite build --watch HMR

# Summary

This RFC proposes to enable hot module reloading (HMR) for Vite's `build --watch` command with a custom web server. The goal is to provide a seamless development experience, allowing developers to see changes reflected in the application without requiring a full page reload.

# Motivation

Vite's `build --watch` command currently rebuilds the application on file changes, but it does not provide HMR capabilities out of the box. This means that developers have to manually reload the page to see the changes. Enabling HMR for `build --watch` with a custom web server would improve the development experience, making it more efficient and productive.

# Detailed Design

To implement this feature, we propose the following:

*   Create a new plugin, `hmr-watch-plugin`, that extends Vite's existing HMR protocol to work with a custom web server.
*   The plugin will handle HMR updates and inject the necessary HMR code into the served files.
*   Vite's core will be modified to support serving HMR code with a custom web server.
*   The plugin will use Vite's plugin API to communicate with the custom web server and update the served files.

The plugin will have the following responsibilities:

*   Handle HMR updates and send them to the custom web server.
*   Inject HMR code into the served files.
*   Handle HMR ping and pong messages from the custom web server.

The custom web server will need to be configured to work with the `hmr-watch-plugin`. This will involve setting up an HMR endpoint that the plugin can use to send updates.

# Drawbacks

The main drawbacks of this proposal are:

*   **Complexity**: Implementing HMR with a custom web server can be complex, especially when dealing with different server configurations and HMR protocols.
*   **Security**: Allowing HMR updates with a custom web server may introduce security risks if not implemented properly.
*   **Performance**: HMR updates can impact performance, especially if the updates are large or frequent.

# Alternatives

The alternatives to this proposal are:

*   **Manual reload**: Developers can manually reload the page to see changes, which is less efficient and productive.
*   **Use a different development server**: Developers can use a different development server that supports HMR out of the box, but this may require additional setup and configuration.
