# RFC: DaemonSet dynamic/scaled resources based on node type

# 1. Summary

This RFC proposes the introduction of dynamic and scaled resource allocation for DaemonSets based on node type in Kubernetes. The goal is to allow DaemonSets to automatically adjust their resource allocation according to the type of node they are running on, improving resource utilization and efficiency.

## 2. Motivation

In current Kubernetes deployments, DaemonSets are used to ensure that a specific pod runs on each node in the cluster. However, nodes in a cluster can have varying resource capacities and characteristics. By dynamically allocating resources to DaemonSets based on node type, we can:

* Improve resource utilization: By allocating resources based on node capacity, we can ensure that DaemonSets use resources more efficiently.
* Enhance scalability: Dynamic resource allocation enables DaemonSets to scale according to node type, ensuring that they can handle changing workloads.
* Simplify management: This feature will simplify the management of DaemonSets in heterogeneous clusters, reducing the need for manual configuration and tuning.

## 3. Detailed Design

The proposed design involves introducing a new API field in the DaemonSet spec to define node-type-based resource allocation policies. This will be achieved using a custom resource definition (CRD) and a controller that watches for changes in node types and updates the DaemonSet resources accordingly.

The new API field will allow users to specify a `nodeResourcePolicy` that defines the resource allocation rules for different node types. For example:

yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: example-daemonset
spec:
  selector:
    matchLabels:
      app: example
  template:
    metadata:
      labels:
        app: example
  nodeResourcePolicy:
    - nodeType: high-performance

          cpu: 2

        cpu: 4
        memory: 8Gi
    - nodeType: standard
      resources:
        requests:
          cpu: 1
          memory: 2Gi
      limits:
        cpu: 2
        memory: 4Gi
The controller will watch for changes in node types and update the DaemonSet resources accordingly. This will involve:

1. Creating a new controller that watches for changes in node types and DaemonSet configurations.
2. Modifying the DaemonSet admission webhook to validate the `nodeResourcePolicy` field.

## 4. Drawbacks

The main drawbacks of this proposal are:

* Increased complexity: Introducing a new API field and controller will add complexity to the Kubernetes codebase.
* Potential performance impact: The controller will need to watch for changes in node types and update DaemonSet resources, which could have a performance impact.

## 5. Alternatives

Alternatives to this proposal include:

* Using existing resource allocation mechanisms, such as ResourceQuotas or LimitRanges, to manage DaemonSet resources.
* Implementing a custom solution using external tools or scripts to manage DaemonSet resources.

However, these alternatives do not provide the same level of integration and automation as the proposed solution.
