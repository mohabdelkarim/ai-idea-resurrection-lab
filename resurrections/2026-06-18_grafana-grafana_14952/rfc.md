# RFC: Support Signal as alert notifier

Summary

    =======

    This RFC proposes adding Signal as a notification channel for alerts in Grafana. The goal is to provide users with an additional option for sending alert notifications, leveraging the Signal messaging platform.

    

    Motivation

    ==========

    Grafana currently supports various notification channels for alerts, including email, Slack, and PagerDuty. However, there is a growing demand for more diverse alerting options, particularly with the increasing adoption of Signal as a communication platform. By adding Signal as a notification channel, users can receive alert notifications in a more convenient and familiar way.

    

    Recent advancements in Signal's API and documentation have made it more feasible to implement Signal as an alert notifier. The introduction of Signal's official API and improvements in Go libraries for interacting with Signal have simplified the integration process.

    

    Detailed Design

    ==============

    To implement Signal as an alert notifier, we will leverage the official Signal API and a Go library that provides a simple interface for sending messages. The feature will involve creating a new notification channel in Grafana's alerting system, configuring Signal as a contact point, and handling authentication and message formatting.

    

    The implementation will require changes to the backend, specifically in the alerting and notification modules. We will follow existing patterns in Grafana's notification system to ensure a seamless user experience.

    

    The proposed implementation will include the following steps:

    1. Add a new notification channel type for Signal in the Grafana backend.

    2. Implement authentication with Signal using a verified phone number and a Signal API token.

    3. Develop a mechanism for formatting alert messages for Signal.

    4. Integrate the Signal notification channel with Grafana's alerting system.

    

    Drawbacks

    =========

    The main drawbacks of this proposal are:

    1. Additional maintenance burden: Supporting Signal as a notification channel will require ongoing maintenance and updates to ensure compatibility with Signal's API.

    2. Potential issues with message delivery: Signal's API may have limitations or issues that could impact message delivery, such as rate limiting or message queuing.

    

    Alternatives

    ===========

    Alternative solutions include:

    1. Using a third-party service that integrates with Signal, such as a webhook or an intermediary platform.

    2. Implementing a custom solution using Signal's API and a scripting language.

    However, these alternatives may introduce additional complexity, costs, or security risks, making the native integration with Signal a more attractive option.

    

    Unresolved Questions

    ===================

    The following questions remain to be addressed:

    1. How will we handle Signal's rate limiting and message queuing policies?

    2. What are the security implications of storing Signal API tokens and verified phone numbers in Grafana?

    3. How will we ensure compatibility with future changes to Signal's API?

    4. What are the performance implications of integrating with Signal's API, particularly for large-scale deployments?
